01

프로젝트 개요

Deep-Briefing이란

Deep-Briefing은 48개의 전문 AI 에이전트가 협력하여 수행하는 전략 리서치 시스템입니다. 4계층 에이전트 계층(PM → Lead → Leaf + Cross-cutting/QA)이 병렬로 작동하며, 파일 시스템 기반 통신으로 프로젝트 의존성 없이 자동 스케일합니다.

핵심 특징
  • Division 병렬 리서치 (핵심 4 + 확장 최대 3)
  • 결정 가능한 분석 (Many Know, Few Decide → We Decide)
  • 4단계 검증 체계 (VL-1~3 + QA)
  • 자동 학습 엔진 (Layer 0~3 지식 시스템)
  • 토큰 최적화 (94% 절감, compile-lead-context.sh)
  • 3-모드 체계 (Auto / Interactive / Team)
02

딥리서치 차별점

차별점 기존 리서치 Deep-Briefing
1. 결정성 많은 정보 제공 ("이런 게 있어요") 명확한 판단 ("이걸 해/하지 마")
2. 수치 검증 단순 인용 출처 추적 + Golden Facts 교차 검증 (VL-3)
3. 반증 포함 확증 편향 (우호적 정보만) Red Team이 핵심 전제 반증 (Phase 3)
4. 반복 심화 한 번에 완성 Phase 1→Sync→Phase 2→Sync 반복 심화
5. 맥락 맞춤 산업 일반론 Client Brief → Division Brief 맞춤 분석
6. 다차원 분석 단순 관점 Market/Product/Capability/Finance 동시 분석 + 교차

주요 설계 원칙
  • MECE 레벨: Division은 "관점(Perspective)"으로, 완벽한 MECE가 아닌 의도적 중첩. Leaf 수준에서만 MECE 보장
  • 파일 시스템 통신: CLI 간 직접 통신 없음. findings/, sync/ 디렉토리와 .done 시그널 경유
  • 동적 스케일: Division별 Leaf 에이전트 자동 스폰, 종료 후 학습 엔진 자동 실행

05

프로토콜 상세

출력 포맷: 4-Layer 피라미드

Layer 0: Claim
        "주장/결론" (경영진 결정 가능한 수준)

Layer 1: Evidence
        "그 근거는?" (분석/인사이트, 출처 명시)

Layer 2: Data
        "데이터는?" (수치, 차트, 구체 사례)

Layer 3: Source
        "어디서?" (URL, 인용문, 신뢰도 스코어)

ID 체계

형식: {Division}{SubDomain}-##

예시: MGE-01 = Market-Geography-EastAsia-01

Golden Facts (SSOT)

  • 파일: findings/golden-facts.yaml
  • 쓰기 권한: fact-verifier만
  • 읽기 권한: 모든 에이전트
  • 역할: 수치 검증의 단일 출처

SCR 스토리라인

  • S (Situation): 현재 상황, 배경, 왜 중요한가
  • C (Complication): 문제, 기회, 긴장, 교착 상태
  • R (Resolution): 해결책, 실행 경로, 기대 효과

Action Title

  • 모든 보고서 섹션 제목은 주장 문장형 (주제형 금지)
  • 나쁜 예: "가격 전략" → 좋은 예: "프리미엄 가격 전략으로 3년간 40% 마진 달성 가능"

Implementation Playbook

template 
각 전략 제안에 포함:
- 담당자: WHO (조직, 역할)
- 마일스톤: WHEN (Q별 일정)
- KPI: HOW MUCH (성공 지표)
- 의존성: WHICH (선행 조건, 리소스)

지식 시스템

4-Layer 지식 구조

Layer위치내용수정 권한
Layer 0 core/knowledge/common-sense.yaml 범용 분석 상식 (소스 신뢰도, 편향 방지, 수치 관례, 함정, 추정 방법론) 시스템
Layer 1 .claude/agents/*.md, leaves/**/*.md 에이전트 역할 지식, 분석 방법론 시스템
Layer 2 domains/{domain}/knowledge/ 산업별 축적 지식 (learned-sources, learned-patterns, learned-terms, learned-frameworks, learned-pitfalls) 학습 엔진 자동
Layer 3 {project}/ 일회성 프로젝트 맥락 (Client Brief, hypotheses, division-briefs) PM

학습 엔진

  • 시점: Division Lead가 Phase 1/2 완료 후 자동 실행
  • 추출: 데이터 소스 신뢰도, 프레임워크 효과성, 분석 패턴, 용어, 함정
  • 저장: {project}/learnings/ → merge-learnings.py로 domains/{domain}/knowledge/에 자동 머지
  • 효과: 프로젝트를 거듭할수록 시스템이 해당 도메인에서 더 똑똑해짐
07

오케스트레이션

Phase 흐름 (총 12단계)

0

Client Discovery + Feasibility Gate

  • 적응형 2-Pass 인터뷰: Pass 1에서 핵심 맥락(배경·문제·의사결정 기준)을 파악하고, Pass 2에서 데이터 갭·우선순위를 심화 질문
  • 산출물: user-profile.yaml (사용자 프로필·선호·제약), Client Brief, Research Plan
  • Research Plan 수립 (활성 Division 결정)
  • Division Briefs 생성 (각 Division의 분석 초점)
0.5

가설 수립 + 1차 데이터 갭 식별

  • Quick Scan (활성 Division 30분)
  • 가설 3~5개 도출
  • 사용자 정렬 (가설 승인)
  • primary_data_gaps 식별
1

Division 병렬 리서치

  • 각 Division Lead가 독립 CLI에서 Leaf 에이전트 스폰
  • 자율 반복 (Leaf 스폰 → 데이터 수집 → VL-1 자가 검증 → Lead로 보고)
  • .done 시그널로 완료 신호
  • 모드 A (tmux 병렬) 또는 모드 B (Agent tool 순차-병렬)
R1

Sync R1: 교차 라우팅 + Tension 식별

  • PM이 Division 출력 수집
  • cross_domain 태깅
  • Tension(T-##) 식별 및 분류
  • fact-verifier VL-3 교차 검증
  • 맥락 체크인: user-profile 기반으로 사용자에게 맥락 있는 확인 질문 수행
2

교차 반영 심화 리서치

  • 각 Division이 다른 Division의 질문 답변
  • 정합성 확보
  • 분석 심화 (증거 강화, 대안 검토)
R2

Sync R2: 교차 정합 + Cross-domain Synthesis

  • Tension 3유형 분류 (data_error / perspective_gap / real_tension)
  • cross-domain-synthesizer → 교차 인사이트 도출
  • Golden Facts 최종 확정
  • 맥락 체크인: 중간 결과를 사용자 맥락에 비추어 검증·방향 확인
3

사고 루프 (Thinking Loop)

  • logic-prober: Why Chain 수직 검증
  • strategic-challenger: 5-레인 도전 (수평 검증)
  • red-team: 적대적 반론 (Auto --deep / Interactive,Team 기본)
  • insight-synthesizer: 도전 결과 통합 + 수렴 판정
3.7

External Review (외부 리뷰)

  • external-reviewer: 자기 비판 + 외부 모델 피드백
  • 약점 체크리스트 5항목: 논리 비약, 증거 부족, 편향, 대안 누락, 실행 가능성
  • 자기 비판(Self-Critique) → 외부 모델 리뷰 → 통합 보강
  • 모드별 차등: Auto(스킵) / Interactive(자기비판만) / Team(외부 모델 포함)
  • 맥락 체크인: user-profile 기반으로 사용자에게 전략적 방향 확인 질문
4

보고서 생성

  • report-writer: SCR 스토리라인 → report-docs.md
  • Trust Badge (검증 배지) 삽입
  • Implementation Playbook 작성
5

QA 자동 수정 루프

  • qa-orchestrator: 4개 내장 검증 + 3개 에이전트 스폰
  • audience-fit-checker (Action Title + SCR)
  • executability-checker (Playbook 검증)
  • report-auditor (논리 감사)
  • 이슈 발견 → report-fixer → 재검증 (최대 3회)
5.5

사용자 피드백 + 부분 재실행

  • 인터랙티브 피드백 분류: L0(오타/포맷) / L1(팩트 수정) / L2(분석 보강) / L3(구조 재설계)
  • 영향 범위 판정 (minimal / division / cross_division)
  • 부분 재실행 (해당 Division 또는 Cross-domain)
6

Post-mortem + 학습 전이 (자동)

  • Leaf 수준 학습 자동 추출
  • domains/{domain}/knowledge/에 자동 머지

3-모드 체계

모드특징질문 예산사용 시기

검증 체계

4단계 검증 (VL-1 ~ VL-3 + QA)

VL-1

Leaf 자가 검증

  • 2개 이상 독립 소스 확보
  • 반증 검토 (반대 의견 존재 여부)
  • Confidence 스코어 태깅
VL-1.5

Lead 삼각 검증

  • Leaf 간 교차 검증 (동일 영역 3개 이상)
  • 스팟체크 3건 이상 (출처 직접 확인)
VL-2

Division 정합성

  • 엔터티 정합성 (동일 기업/시장을 같은 이름으로)
  • 시점 정합성 (과거/현재/미래 명확 구분)
  • 정의 일관성 (지표 산출 방식 동일)
VL-3

교차 검증 (fact-verifier)

  • Division 간 모순 탐지
  • Golden Facts 도출 (수치 SSOT)
  • 신뢰도 스코어 매기기

QA Phase 5: 보고서 검증 (8+1단계)

  • 내장 4단계: 구조/링크/포맷/메타데이터
  • 외장 3단계: audience-fit-checker / executability-checker / report-auditor
  • 수정 루프: report-fixer (최대 3회)
  • 합격 기준: Critical/Major 이슈 0건
