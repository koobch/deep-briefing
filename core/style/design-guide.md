### 간격 토큰 시스템 (Spacing Scale)

> 모든 간격(gap, margin, padding)은 반드시 토큰을 사용한다. px 직접 지정 금지.

| 토큰 | 값 | 용도 | 예시 |
|------|-----|------|------|
| `--sp-2xs` | 2px | border offset, 미세 간격 | 쉐브론 사이 |
| `--sp-xs` | 4px | Big Number↔라벨, 같은 블록 내 텍스트 | metric-value ↔ metric-label |
| `--sp-sm` | 8px | 아이콘↔텍스트, 도형↔설명, 불릿 내부 | rank-icon ↔ rank-label |
| `--sp-md` | 12px | 불릿 항목 간, 컨테이너 내부 상하 패딩 | case-row-body padding-top |
| `--sp-lg` | 16px | 문단 간, 플로우 그리드 간격 | panel-text 문단 gap |
| `--sp-xl` | 20px | 행 좌우 패딩, 컬럼 내부 간격 | case-row-body padding-left |
| `--sp-2xl` | 24px | 대섹션 간 간격 | col-primary gap |
| `--sp-3xl` | 32px | 슬라이드 주요 구역 간 | footer padding-bottom |
| `--sp-4xl` | 44px | 헤더→본문 | slot-header → slot-body |

**강제 규칙**:
- 인라인 `gap:Npx`, `margin:Npx`, `padding:Npx` 사용 **금지** → `var(--sp-*)` 사용
- 토큰에 없는 값(예: 14px, 36px)은 가장 가까운 토큰으로 정규화 (14→16=--sp-lg, 36→32=--sp-3xl)

### 행간(Line Height) 3단계

> line-height는 3단계만 사용한다. 중간값(1.25, 1.35 등) 사용 금지.

| 단계 | 값 | 적용 대상 |
|------|-----|---------|
| **tight** | 1.15 | Action Title (`--fs-title`), Big Number (`--fs-big`), Rank Number (`--fs-rank`) |
| **normal** | 1.3 | 본문, 패널, 라벨, Source, Callout, 불릿, 표 — **`:lang(ko)` 기본값** |
| **relaxed** | 1.45 | Sub-question (`--fs-subq`) — 선택적, 기본은 normal |

**강제 규칙**:
- 인라인 `line-height:N` 사용 시 위 3단계 값만 허용
- 1.2, 1.25, 1.3, 1.35, 1.4 등 중간값 사용 **금지**

### P6 상세: 엣지 정렬 원칙

**문제 패턴**: container에 `padding-left/right` + `background`를 동시 적용하면,
배경은 padding 포함 전체를 채우지만 자식은 padding 안쪽에서 시작 → 인접 행과 어긋남

**해결 패턴**:
1. 배경색이 있는 행의 container는 **좌우 padding을 0**으로 설정
2. 배경이 전폭으로 깔림
3. 콘텐츠의 좌우 인덴트는 **내부 요소에서 통일 토큰(`--sp-xl` = 20px)**으로 적용
4. 결과: 배경 행이든 일반 행이든 콘텐츠 시작/끝점이 동일

**금지**: container에 padding-left/right + background 동시 적용
**적용 유형**: Case Study, Table/List, Exec Summary, **Big Number (우측 영역)**, **Ranked (ranked-bar/icons/detail)**, **Process (쉐브론/하단 detail)** 등 배경색 행이 있는 모든 레이아웃

### P7 상세: 행 간 Grid 기준선 공유

**문제 패턴**: 인접 행에서 metrics(3-column)와 flow-box(3-column)가 각각 다른 padding을 가지면 열 경계가 어긋남

**해결 패턴**:
1. 같은 body 영역 내 인접 행의 grid 콘텐츠는 **동일한 grid-template-columns** 사용
2. grid 컨테이너의 **좌우 padding은 0**
3. arrow 등 장식 요소는 **::after pseudo-element**로 배치
4. 결과: metrics의 N등분 경계와 flow 박스의 N등분 경계가 정확히 일치

**적용 유형**: Case Study, 향후 multi-row grid 패턴 전체

### 도형-텍스트 간격 규격

> 도형(아이콘, 박스, 쉐브론)과 텍스트 사이 간격은 역할별로 고정한다.

| 관계 | 간격 | 토큰 | 예시 |
|------|------|------|------|
| Big Number ↔ 라벨 | 4px | `--sp-xs` | "80%" ↔ "디지털 마케팅 ROI" |
| 라벨 ↔ 설명 | 4px | `--sp-xs` | "Capex 최적화" ↔ "연간 투자 예산 대비 절감 효과" |
| 아이콘 ↔ 텍스트 | 8px | `--sp-sm` | 🎯 ↔ "Impact 성과" |
| 플로우 박스 ↔ 설명 | 8px | `--sp-sm` | [오디언스 엔진] ↔ "고객 세그먼트 분석" |
| 불릿 항목 간 | 12px | `--sp-md` | "• 항목1" ↔ "• 항목2" |
| 문단 간 (패널) | 16px | `--sp-lg` | 문단1 ↔ 문단2 |
| 행 내부 상하 패딩 | 12px | `--sp-md` | case-row, table-row |
| 행 좌우 패딩 | 20px | `--sp-xl` | case-row-body, table-row-content |
| Big Number 그룹 간 | 32px | `--sp-3xl` | "48%" 그룹 ↔ "-9pp" 그룹 |
| Table/List 라벨 아이콘↔텍스트 | 8px | `--sp-sm` | 🔧 ↔ "삽질의 역설" (아이콘 flex-shrink:0, 텍스트 수직 중앙) |

**Big Number 그룹 정의**: Big Number + 라벨 + 설명(선택) = 1그룹. 그룹 내부는 기존 규칙(4px), 그룹 간은 `--sp-3xl`(32px).

**도형 containment 규칙**:
- 모든 도형은 부모 컨테이너 내에 완전히 포함되어야 한다
- `overflow: hidden`으로 잘리는 것이 아닌, 크기/여백으로 자연스럽게 포함
- 플로우 박스는 `width: 100%`로 컬럼 폭에 맞춤
- 아이콘은 고정 크기(72px) + 중앙 정렬

### Scatter 버블 크기 패턴

> 데이터 시각화에서 버블 크기가 데이터값(예: 매출 규모)을 나타낼 때 사용

```html
<!-- CSS custom property로 버블 크기 지정 -->
<div class="scatter-point" style="left:12%;bottom:25%;--bbl:56px;background:var(--c-accent)">A사</div>
```

- CSS에서 `width: var(--bbl, var(--fs-title)); height: var(--bbl, var(--fs-title))` 으로 소비
- `--bbl` 미지정 시 기본 크기 `--fs-title` (48px) 적용
- **규칙**: 인라인에 `width:Npx;height:Npx` 대신 `--bbl:Npx`로 토큰화

## D. 콘텐츠 구조 & 작성법

### Head Message (Action Title) 작성법

**규칙**: 모든 슬라이드 제목은 **주장형 문장**(Action Title)이어야 한다.

| 패턴 | 예시 | 용도 |
|------|------|------|
| [주체]는 [행동]으로 [결과]를 달성한다 | "비용 재설계와 성장 투자 병행으로 수익성·매출 질 동시 개선 가능" | Exec Summary |
| [수치]%의 [대상]이 [상태]이다 | "임원의 86%가 2025년 AI 투자를 계획하고 있다" | Insight+Chart |
| [대상]은 [요인]을 [수준]으로 우려한다 | "아시아태평양 임원들은 수출 둔화가 성장을 저해할 수 있다고 우려한다" | Insight+Chart |
| [방법]으로 [목표]를 확보한다 | "설계-가치 최적화 접근법으로 경쟁력 있는 포트폴리오를 확보한다" | Process |

**금지 패턴**:
- ❌ 주제형: "비용 분석", "시장 현황", "AI 전략 개요"
- ❌ 모호한 표현: "다양한 방면에서 검토", "여러 요인이 작용"
- ❌ 질문형 제목: "어떻게 해야 하는가?" (질문은 Sub-question에만)

### Action Title 마스터 테이블

유형별 맞춤 공식. 슬라이드 유형이 결정되면 해당 공식에 맞춰 Action Title을 작성한다.

| 유형 | Action Title 공식 | 예시 |

| Process | [방법]으로 [목표]를 확보한다 | "설계-가치 최적화 접근법으로 경쟁력 있는 포트폴리오를 확보한다" |

### Sub-question
- **위치**: 본문 영역 최상단 (Action Title 바로 아래)
- **형식**: "~은 무엇인가?", "~에 어떤 영향을 미치는가?"
- **폰트**: `--fs-subq` (32px), bold, 다크 초록 (`#03522d`)
- **용도**: 차트나 데이터 앞에 배치하여 독자에게 "이 데이터가 답하는 질문"을 명시

### So-What / Why-So 논리 구조

```
[Action Title] ← So-What (이 슬라이드의 핵심 주장)
    ↑
[Sub-question] ← 독자의 궁금증을 명시
    ↑
[Evidence 1] [Evidence 2] [Evidence 3] ← Why-So (주장을 뒷받침하는 근거)
    ↑
[Data / Chart] ← 근거의 데이터 기반
    ↑
[Source] ← 데이터 출처
```

**적용 규칙**:
- Action Title은 반드시 본문 내용에서 도출된 So-What이어야 함
- 본문의 모든 문단은 Action Title을 뒷받침하는 Why-So여야 함
- 관련 없는 내용은 다른 슬라이드로 분리

### So-What / Why-So 검증 체크리스트

슬라이드 작성 후 반드시 다음 5개 항목을 검증한다:

1. **역순 테스트(Reverse Test)**: Action Title을 가리고 본문만 읽었을 때, 독자가 같은 결론(So-What)을 도출할 수 있는가?
   - Yes → 논리 구조 성립
   - No → Evidence가 부족하거나 논리 비약이 있음 → 보강 필요
   - 예시: 본문에 "AI 투자 계획 86%, 생산성 향상 기대 72%" 데이터가 있으면 → "임원 다수가 AI 투자를 적극 추진 중"이라는 결론이 자연스럽게 도출되어야 함

2. **관련성 테스트(Relevance Test)**: 본문의 모든 문단이 Action Title의 Why-So인가?
   - 관련 없는 문단 발견 → 다른 슬라이드로 분리
   - 1슬라이드 = 1메시지 원칙 위반 여부 점검
   - 예시: "AI 투자 계획" 슬라이드에 "공급망 이슈" 문단이 혼재 → 분리 대상

3. **질문-답변 테스트(Q&A Test)**: Sub-question이 있는 경우, 차트/데이터가 그 질문에 정확히 답하는가?
   - Sub-question과 차트의 데이터가 미스매치 → Sub-question 수정 또는 차트 변경
   - 예시: Sub-question "어떤 산업이 AI를 가장 적극 도입하는가?" → 차트가 산업별 도입률 비교를 보여야 함 (연도별 추이 차트는 미스매치)

4. **Callout 정합 테스트(Callout Coherence Test)**: Callout이 차트 데이터에서 직접 도출 가능한가?
   - Callout에 차트에 없는 정보가 포함 → 차트 보강 또는 Callout 수정
   - 예시: 차트에 5개 산업 비교 → Callout "금융이 제조 대비 2.3배 높은 도입률" → 차트에서 두 수치가 확인 가능해야 함

5. **패널 보완 테스트(Panel Complement Test)**: 우측 패널 내용이 차트가 보여주지 못하는 맥락/해석인가?
   - 패널이 차트 내용을 단순 반복 → 맥락/배경/전략적 시사점으로 교체
   - 예시: 차트 "금융 42%, 제조 18%" → 패널에서 "금융이 높은 이유는 **규제 기관의 디지털 전환 압력**과 레거시 시스템 교체 수요가 겹쳤기 때문"으로 배경을 설명

> **전수 검증 원칙**: 위 5개 테스트 중 하나라도 No이면 슬라이드를 수정한 후 재검증한다.

### 불릿 포인트
- **기호**: `•` (초록색 `#28ba74`)
- **들여쓰기**: 20px (1단계만, 2단계 이상 지양)
- **구조**: 각 불릿은 1~2문장, 핵심 구절에 **bold** 적용
- **개수**: 슬라이드당 3~5개 (7개 초과 금지)

**Exec Summary 불릿 특수 규칙**: Exec Summary 유형에서 불릿(•)은 리드 문장과 **동일 x좌표**에서 시작. 불릿 기호는 좌측으로 돌출(hanging indent) 허용. 이는 BCG 레퍼런스의 Exec Summary 패턴을 따른 것.

### 불릿 논리 구조

불릿은 단순 나열이 아니라, 다음 중 하나의 논리 구조를 따라야 한다:

| 구조 | 설명 | 예시 |
|------|------|------|
| **MECE 분류** | 상호 배제·전체 포괄(Mutually Exclusive, Collectively Exhaustive) | 지역별(북미/유럽/아시아), 기능별(마케팅/운영/재무) |
| **인과순** | 원인 → 결과 → 시사점 | 관세 상승 → 마진 압박 → 공급망 다변화 필요 |
| **시간순** | 과거 → 현재 → 미래 | 2023년 도입 → 2024년 확산 → 2025년 전사 적용 |
| **중요도순** | 1순위 → 2순위 → 3순위 | 비용 관리 > 성장 투자 > 인력 재편 |

**선택 기준**:
- 데이터가 카테고리별로 나뉘면 → MECE 분류
- 원인과 결과가 있는 스토리면 → 인과순
- 변화 추이를 설명하면 → 시간순
- 우선순위/랭킹이 있으면 → 중요도순

**금지**: 관련성 없는 항목의 임의 나열 (예: "AI 투자 증가 / 인플레이션 우려 / 조직 개편" — 하나의 논리 축이 없음)

### 강조 텍스트 규칙
- **Bold 적용**: 문장 내 핵심 구절 1~2개에만
- **예시**: "임원의 **86%가 AI 투자를 계획**하고 있다"
- **금지**: 문장 전체 bold, 연속 3개 이상 bold 구절

### Callout 박스
- **배경**: `#d0f5e2` (연초록)
- **위치**: 차트 아래 또는 본문 하단
- **용도**: 핵심 시사점을 한 문장으로 요약
- **구조**: bold 키워드 + 설명

### 윗첨자/아랫첨자
- **각주 번호**: 윗첨자로 `1.`, `2.` 표기, Source 영역에서 해설
- **단위**: %, $, ₩ 등은 숫자와 함께 인라인
- **기준 시점**: 괄호로 명시 (예: "2024년 4분기")

### 우측 패널(Secondary Panel) 작성법

패널은 차트가 보여주지 못하는 **맥락·해석·시사점**을 제공한다.

| 구분 | 차트 영역 | 패널 영역 |
|------|---------|---------|
| 역할 | 데이터(What) | 맥락(Why/So What) |
| 내용 | 수치, 비율, 추세 | 배경, 원인, 전략적 의미 |
| 형식 | 시각화, 라벨 | 서술형 문단, bold 핵심구 |
| 최소 밀도 | 차트 항목 4개 이상 | 3문단 이상, 각 문단에 bold 1개 |

**패널 문단 구조**: 각 문단은 다음 형태를 따른다:
> 배경/원인 1~2문장 + **bold 핵심 시사점** + 설명 1문장

**예시** (Insight+Chart 슬라이드의 우측 패널):
> 글로벌 공급망 불확실성이 지속되면서 기업의 전략적 대응이 분화되고 있다. **선도 기업의 73%는 이미 공급망 이중화를 완료**했으며, 이는 후발 기업(28%) 대비 2.6배 높은 수준이다. 이러한 격차는 향후 3년간 원가 경쟁력에서 결정적 차이로 나타날 전망이다.

**패널 작성 시 금지 사항**:
- 차트 데이터를 텍스트로 반복 서술 (예: 차트에 "27%" 표시 → 패널에서 "27%가 인플레이션을 우려한다" 반복 금지)
- 차트와 무관한 별도 주장 삽입
- 출처 없는 수치 추가 (패널의 추가 수치도 golden-facts.yaml 기반이어야 함)

**패널이 제공해야 하는 3가지 가치**:
1. **배경(Context)**: 왜 이런 데이터가 나왔는지 — 시장 환경, 정책 변화, 기술 트렌드
2. **해석(Interpretation)**: 이 수치가 의미하는 바 — 산업 평균 대비, 과거 추세 대비
3. **시사점(Implication)**: 의사결정자가 취해야 할 행동 — 전략적 방향, 리스크 대응

## D+. deep-briefing 리서치 → 슬라이드 변환 규칙

### 4-Layer 피라미드 → 슬라이드 매핑

| Layer | 역할 | 슬라이드 요소 |
|-------|------|-------------|
| Layer 0 (Claim) | 핵심 주장 | **Action Title** (헤더) |
| Layer 1 (Evidence) | 근거/논증 | 본문 텍스트, 불릿 포인트, 우측 패널 |
| Layer 2 (Data) | 데이터/수치 | 차트, Big Number, 표, KPI |
| Layer 3 (Source) | 출처 | Source 푸터, 각주 |

### Division Findings → 슬라이드 유형 선택

| 발견 유형 | 추천 슬라이드 유형 | 예시 |
|----------|------------------|------|
| 핵심 수치 1개 | Big Number | "시장 규모 $4.2B" |
| 비율/구성 데이터 | Insight+Chart (파이/도넛) | "매출 비중: A 40%, B 35%, C 25%" |
| 순위/우선순위 | Ranked (#1~N) | "임원 우선순위: 공급망 > 포트폴리오 > 인력" |
| 시계열 변화 | Insight+Chart (바/라인) | "매출 추이 2020~2025" |
| 프로세스/방법론 | Process | "3단계 최적화: 기준→아이디어→실행" |
| 전략 제안 | Exec Summary | "3가지 전략 축: 비용/성장/실행" |
| 경쟁 포지셔닝 | Matrix (2×2) | "시장 점유율 vs 성장률 매트릭스" |
| 전후 비교 | Comparison (Before/After) | "도입 전 vs 도입 후 비용 구조" |
| 조직/분류 체계 | Tree / Hierarchy | "사업부별 역할 분담 구조" |
| 사례 실증 | Case Study | "A사 도입 사례: 20% 비용 절감" |

### golden-facts.yaml → 데이터 시각화 결정 트리

```
단일 핵심 수치? ─── Yes → Big Number
  │
  No
  │
비율/구성? ──────── Yes → 파이/도넛 차트
  │
  No
  │
시계열? ─────────── Yes → 라인/바 차트
  │

순위/비교? ──────── Yes → 수평 막대 or Ranked
```

### SCR 스토리라인 → 슬라이드 시퀀스

| SCR 단계 | 슬라이드 구성 | 유형 |
|----------|-------------|------|
| Situation | 현황 분석 1~2장 | Insight+Chart |
| Complication | 문제 제기 1~2장 | Comparison, Big Number |
| Resolution | 해결책 2~3장 | Process, Framework, Exec Summary |

### Action Title 작성 공식

| 공식 | 예시 |
|------|------|
| `[주체]는 [행동]으로 [결과]를 [달성]한다` | "비용 문화 정착으로 11% 추가 절감을 달성한다" |
| `[수치]%의 [대상]이 [상태]이며 [시사점]이다` | "48%만 목표 달성, 미달 기업은 TSR 9pp 저조" |
| `[대상]은 [요인]을 [최우선]으로 추진한다` | "임원들은 공급망 최적화를 최우선으로 추진한다" |

> **전체 유형별 공식은 D섹션 "Action Title 마스터 테이블" 참조.** 위 3가지는 deep-briefing 리서치 결과물에서 가장 빈번하게 사용되는 핵심 공식이다.

| 공식 | 주 사용 Phase |
|------|-------------|
| `[주체]는 [행동]으로 [결과]를 [달성]한다` | Phase 3~4 (Resolution) |
| `[수치]%의 [대상]이 [상태]이며 [시사점]이다` | Phase 1~2 (Situation/Complication) |
| `[대상]은 [요인]을 [최우선]으로 추진한다` | Phase 2 (Complication/Resolution) |

### SCR 단계별 Action Title 가이드

SCR 스토리라인의 각 단계에서 적합한 Action Title 패턴이 다르다:

| SCR 단계 | 권장 패턴 | 예시 |
|----------|---------|------|
| Situation (현황) | `[수치]%의 [대상]이 [상태]이다` | "글로벌 기업의 78%가 AI 도입을 완료하거나 진행 중이다" |
| Complication (문제) | `[대상]은 [양극화/격차/위험]을 보인다` | "선도 기업과 후발 기업 간 생산성 격차가 2.4배로 확대되고 있다" |
| Resolution (해결) | `[방법]으로 [목표]를 [달성]한다` | "3단계 역량 구축 프로그램으로 AI 내재화 격차를 해소한다" |

### Division Findings → Action Title 변환 프로세스

리서치 결과물에서 Action Title을 도출하는 단계별 절차:

1. **Claim 추출**: findings의 Layer 0(Claim)에서 핵심 주장을 식별
2. **유형 매칭**: 주장의 성격에 맞는 슬라이드 유형을 위 "Division Findings → 슬라이드 유형 선택" 테이블에서 결정
3. **공식 적용**: 결정된 유형에 맞는 Action Title 공식을 D섹션 마스터 테이블에서 선택
4. **수치 삽입**: golden-facts.yaml의 검증된 수치를 공식에 삽입
5. **검증**: D섹션 "So-What / Why-So 검증 체크리스트"의 역순 테스트 실행

**예시 -- Market Division Finding 변환**:
- Finding: "시장 규모 $4.2B, CAGR 12.3%, 아시아 비중 38%"
- Claim: "아시아 시장이 가장 빠르게 성장하며 글로벌 비중을 확대 중"
- 유형: Insight+Chart (비율/구성 + 시계열)
- Action Title: "아시아 시장의 38% 비중은 CAGR 12.3%로 글로벌 평균을 상회하며 2028년 45%까지 확대 전망이다"

## E. 데이터 시각화

### 차트 유형별 규칙

| 차트 | 용도 | 색상 | 크기 |
|------|------|------|------|
| 수평 막대 | 항목 비교 (3~8개) | 초록 계열 그라데이션 | 바 높이 28~36px, gap 8~12px |
| 수직 막대 | 시계열 | 초록 + 보조색 | — |
| 파이/도넛 | 비율 (2~5개) | 초록~회색 계열 | 직경 200~300px |
| 라인 | 추세 | 초록 실선 | — |

### 차트 내 텍스트
- **데이터 라벨**: `--fs-chart-val` (28px), bold
- **축 라벨**: `--fs-label` (22px), regular
- **범례**: 차트 하단 또는 우측, 12~14px

### 차트 배치
- 좌측 컬럼에 배치 (65% 영역)
- 차트 위에 Sub-question, 아래에 Callout
- 차트 영역은 `flex:1`로 가용 공간 채움

### 차트 내 텍스트 계층 (우선순위 순)

| 순위 | 요소 | CSS 변수 | 크기 | 스타일 | 역할 |
|------|------|---------|------|--------|------|
| 1 | 데이터 값 | `--fs-chart-val` | 28px | bold | 가장 먼저 눈에 들어와야 함 |
| 2 | 항목 라벨 | `--fs-label` | 22px | regular | 값이 무엇인지 설명 |
| 3 | 범례 | `--fs-source` | 18px | regular | 색상/패턴 구분 |
| 4 | 축 라벨 | `--fs-source` | 18px | regular | X/Y축 의미 |

**배치 원칙**: 데이터 값은 바 끝/파이 옆에 직접 표기 (별도 축 참조 불필요하게)

### 수평 막대 데이터 라벨 배치 규칙
- 바 폭이 라벨 텍스트 폭의 1.5배 이상 → **바 내부 우정렬** (white 텍스트)
- 바 폭이 라벨 텍스트 폭의 1.5배 미만 → **바 우측 외부** `--sp-sm`(8px) 간격
- 외부 라벨은 모두 **동일한 x좌표**(가장 긴 바의 우측 끝 + `--sp-sm`)에 좌정렬
- 이 규칙은 수평 막대 차트에만 적용 (수직 막대는 상단 배치)

### 차트 표준 색상 팔레트

| 팔레트 | 용도 | 색상 (최대 5색) |
|--------|------|---------------|
| **순서형** | 순위, 단일 시리즈 | `#04512c` → `#187955` → `#28b974` → `#74e0ac` → `#c7c7c7` |
| **비교형** | Pre/Post, 지역 비교 | `#285f7d`(Pre) + `#3a7a9e`(Post) |
| **대비형** | 긍정/부정, 성공/실패 | `var(--c-accent-dark)` + `var(--c-danger)` |
| **카테고리형** | 3~5개 항목 구분 | `#04512c`, `#6d352c`, `#285f7d`, `#8a6d3b`, `#c7c7c7` |

**규칙**:
- 같은 슬라이드 내 차트 색상은 3색 이내 권장
- 순서형 팔레트는 상위→하위로 다크→라이트 그라데이션
- 긍정은 초록 계열, 부정은 빨강 계열 (`--c-danger`)
- 회색(`#c7c7c7`)은 "기타" 또는 "하위" 항목에만 사용

### 차트 선택 의사결정 트리 (상세)

데이터를 받았다. 어떤 차트를 선택할 것인가?

1단계: 데이터 수 판단
├─ 핵심 수치 1~2개 → **Big Number** (56~72px)
└─ 다수 항목 → 2단계로

2단계: 데이터 관계 판단
├─ 카테고리 비교 (3~8개 항목)
│  ├─ 항목 라벨이 긴 텍스트 → **수평 막대** (라벨 좌측)
│  └─ 항목 라벨이 짧은 (연도 등) → **수직 막대**
├─ 시계열 변화
│  ├─ 항목 4개 이하 (연도) → **라인 차트**
│  ├─ 항목 5개 이상 → **그룹 수직 막대**
│  └─ Pre/Post 비교 → **그룹 수평 막대** (2색)
├─ 전체 대비 비율
│  ├─ 항목 2~4개 → **도넛 차트** (직경 200~300px)
│  └─ 항목 5개 이상 → **수평 막대** (전체를 100%로)
├─ 양극화/긍정-부정 대칭 → **Divergent Bar** (좌우 대칭)
├─ 2차원 포지셔닝 → **산점도/버블**
├─ 다차원 격자 → **히트맵** (색상 코딩)
└─ 프로세스 진행률 → **게이지/진행바** (도넛 변형)

### 차트-Callout-패널 3요소 관계

하나의 Insight+Chart 슬라이드에서 3요소는 각각 다른 역할을 수행한다:

| 요소 | 역할 | 질문 | 형식 |
|------|------|------|------|
| **차트** | 데이터 제시 (What) | "수치가 어떻게 분포/변화하는가?" | 시각화 + 데이터 라벨 |
| **Callout** | 핵심 발견 (So-What) | "이 데이터에서 가장 중요한 발견은?" | 한 문장, bold 핵심구 |
| **패널** | 맥락 해석 (Why/What Next) | "왜 이런 결과가? 전략적 의미는?" | 3~5문단, 서술형 |

**금지 패턴**:
- Callout이 차트 값을 단순 반복: ❌ "27%가 인플레이션을 우려한다"
- Callout이 올바른 So-What: ✅ "대선 이후 경제 불확실성에 대한 우려가 전 지역에서 증가했다"
- 패널이 차트를 텍스트로 다시 설명: ❌ → 패널은 배경/원인/시사점만

| 요소 | 스타일 |
|------|--------|
| 헤더 행 | 배경 `#04512c` (다크 초록), 텍스트 흰색, bold |
| 본문 행 | 배경 교대 (`#ffffff` / `#f5f5f5`), 텍스트 검정 |
| 테두리 | 1px `#b1b2b1` (연회색), 헤더 하단만 2px |

| 강조 셀 | 배경 `#d0f5e2` 또는 bold 텍스트 |

### Table vs Table/List 구분

| 구분 | Table (정형) | Table/List (행 나열) |
|------|------------|-------------------|
| 용도 | 정형 데이터 비교 (숫자/텍스트) | 이슈/과제/요인 나열 |
| 구조 | 헤더 행 + 데이터 행 | 마커(a~f) + 제목 + 설명 |
| 레이아웃 | 유형 J.10 외 — grid-template-columns | 유형 J.10 전용 — layout-table-list |
| 정렬 | 텍스트 좌, 숫자 우 | 제목 좌, 설명 좌 |

### Table/List 각 행 콘텐츠 작성법

각 행은 다음 3요소를 포함해야 한다:

| 요소 | 역할 | 형식 |
|------|------|------|
| **현상** | "무엇이 문제인가?" | bold 핵심 문장 (첫 문장) |
| **원인** | "왜 문제인가?" | 설명 1문장 (인과 관계) |
| **필요 행동** | "무엇을 해야 하는가?" | 설명 1문장 (방향 제시) |

**예시** (ai-engineering-p05 기준):
- [a] "삽질의 역설" | **GenAI는 "재미있는" 코딩을 자동화**하지만(현상) 테스트, 보안, 요구사항은 병목으로 남아(원인) — 생산성 효과의 완전 실현에는 마인드셋 전환이 필요하다(필요 행동)

### 행 수별 레이아웃 조정

| 행 수 | 조정 | font-size |
|-------|------|----------|
| 4~5행 | 여유 — 각 행 설명 2~3줄 | `--fs-body` (22px) |
| 6~7행 | 표준 — 각 행 설명 1~2줄 | `--fs-body` (22px) |
| 8행 | compact — 각 행 설명 1줄 | `--fs-body` (22px), 행간 축소 금지 |
| 9행+ | 금지 — 2장으로 분할 | — |

### 표지 (Cover)
- 전체 배경: 사진 또는 다크 그라데이션
- 중앙 또는 좌측: 제목 텍스트 (큰 폰트)
- 하단: 날짜/저자

### 뒷표지 (Back Cover)
- 텍스트 최소화
- 로고/연락처/copyright

### Agenda 활성 전환 규칙

보고서 내 Agenda 슬라이드는 SCR 스토리라인의 **섹션 전환을 명시**하는 역할이다.

| 삽입 위치 | 활성 항목 | 역할 |
|----------|---------|------|
| 보고서 시작 (2번째 장) | 1번 항목 활성 | "지금부터 Situation을 다룬다" |
| Complication 시작 전 | 2번 항목 활성 | "Situation에서 Complication으로 전환" |
| Resolution 시작 전 | 3번 항목 활성 | "Complication에서 Resolution으로 전환" |

**구현**: `.agenda-item--active`와 `.agenda-item--inactive` 클래스 전환
**규칙**: Agenda 항목 텍스트는 해당 섹션의 핵심 주제를 **주제형**으로 기술 (유일한 예외 — Action Title 주장형 규칙 미적용)

### Cover-본문-Back Cover 논리 연결

| 위치 | 요소 | 논리적 역할 |
|------|------|-----------|
| Cover | 보고서 제목 | 보고서 전체의 **So-What 예고** — 읽기 전 핵심 메시지 전달 |
| Exec Summary | Action Title | Cover 제목의 **구체화** — 3축(S/C/R)으로 분해 |
| 본문 | 각 장 Action Title | Exec Summary 3축의 **Why-So 증명** |
| Back Cover | 마무리 | 행동 촉구(CTA) — "Read more", 연락처, 관련 보고서 |

**Cover 제목 작성법**:
- Exec Summary의 Action Title을 한 문장으로 압축
- 예: Exec Summary "비용 재설계와 성장 투자 병행으로 수익성·매출 질 동시 개선 가능" → Cover "Cost Excellence: Building Resilience Through Sustainable Cost Transformation"

### Back Cover 상세 구조

| 요소 | 내용 | 스타일 |
|------|------|--------|
| 로고 | 조직 로고 | 좌상단 또는 중앙 |
| CTA 텍스트 | "더 자세한 인사이트를 확인하세요" | 큰 폰트, 중앙 |
| 관련 보고서 | 링크 2~3개 | 텍스트 링크 형태 |
| 연락처 | 담당자/이메일 | 작은 폰트, 하단 |
| Copyright | "© 2025 [조직]" | 12px, 하단 우측 |

## I. Source & 별첨

### Source 텍스트
- **위치**: 슬라이드 하단 좌측 (푸터)
- **폰트**: 18px, `#7f7f7f`
- **형식**: "Source: [출처] ([날짜]); [분석 주체]"
- **각주**: "주: [설명]" + 줄바꿈 + "Source: ..."

### 페이지 번호
- **위치**: 푸터 우측
- **폰트**: 22px, `#898989`

### Copyright
- **위치**: 우측 세로 (writing-mode: vertical-rl)
- **폰트**: 12px, `#575757`

### Source 작성법 상세

#### 데이터 유형별 형식

| 데이터 유형 | Source 형식 | 예시 |
|-----------|-----------|------|
| 설문 조사 | "Source: [설문명] (N=[표본 수]) [기간]; [분석 주체]" | "Source: BCG 글로벌 임원 설문 (N=570) 2024년 4분기; BCG 분석" |
| 공시/통계 | "Source: [기관] [데이터셋명] ([기간])" | "Source: World Bank GDP 데이터 (2020~2024)" |
| 기업 보고 | "Source: [기업] [보고서명] ([연도]); [분석 주체] 분석" | "Source: Apple 10-K (2024); BCG 분석" |
| 산업 리포트 | "Source: [기관] [리포트명] ([연도])" | "Source: Gartner Hype Cycle for AI (2024)" |
| 내부 분석 | "Source: [분석 주체] 분석" | "Source: BCG 분석" |
| 다중 출처 | "Source: 1. [출처1]; 2. [출처2]; [분석 주체] 분석" | "Source: 1. 금융감독원 공시 (2024); 2. 한국은행 경제통계 (2024); BCG 분석" |

#### 각주 작성법

| 요소 | 형식 | 예시 |
|------|------|------|
| 본문 내 표기 | 윗첨자 번호 (1., 2.) | "아시아태평양\u00b9" |
| Source 영역 | "주: [번호]. [설명]" 후 Source | "주: 1. 아시아태평양은 중국 제외\nSource: BCG..." |
| 약어 풀이 | Source 앞 "주:" 라인 | "주: RTB: 실시간 입찰(Real-Time Bidding)" |

#### Source 품질 체크리스트
- [ ] 모든 차트/수치에 Source가 있는가?
- [ ] 설문 데이터에 표본 수(N=)와 기간이 있는가?
- [ ] "BCG 분석"만 있는 경우, 원시 데이터의 출처도 명시했는가?
- [ ] 각주 번호가 본문과 Source 영역에서 일대일 매칭되는가?

### J.1 Cover (표지)
- **레이아웃**: 전폭, 다크 배경 + 사진 오버레이
- **필수 요소**: 보고서 제목 (큰 폰트 40~56px), 부제/날짜, 로고
- **Action Title**: 보고서 핵심 테마를 한 문장으로 (예: "AI-First Companies Win the Future")
- **deep-briefing 입력**: 리서치 주제 + 프로젝트명
- **논리 흐름**: 보고서 테마 → 제목+부제로 핵심 메시지 전달 → 독자의 기대 설정

### J.2 Agenda (목차)
- **레이아웃**: 좌 40% 사진패널 + 우 60% 목록
- **필수 요소**: 체크마크 아이콘 (활성 초록/비활성 회색) + 섹션 제목 3~5개
- **작성법**: 각 섹션 제목은 주장형이 아닌 주제형 허용 (유일한 예외)
- **deep-briefing 입력**: SCR 스토리라인 섹션 구분
- **논리 흐름**: 섹션 목록 표시 → 현재 위치 강조(초록) → 보고서 전체 구조 환기

### J.3 Introduction (도입)
- **레이아웃**: 좌 65% 텍스트 + 우 35% 이미지패널
- **필수 요소**: 제목(소제목 스타일), 본문 2~3문단, 불릿 3~5개, bold 핵심구
- **Action Title**: "Introduction" 라벨 + 핵심 메시지
- **작성법**: 보고서의 문제 정의, 범위, 접근법을 명시. "이 보고서에서 우리는 ~를 다룬다"
- **deep-briefing 입력**: 00-client-brief.md의 배경/목적
- **논리 흐름**: 배경/범위 서술 → 불릿으로 핵심 포인트 구조화 → Action Title이 도입부 결론
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.4 Insight + Chart (데이터 인사이트)
- **레이아웃**: 좌 65% (Sub-q + 차트 + Callout) + 우 35% 패널
- **필수 요소**: Sub-question, 차트(바/파이/라인), Callout, 패널 텍스트 3~5문단
- **Action Title**: "[대상]은 [상태]이며 [시사점]이다" 패턴
- **작성법**:
  - Sub-question이 차트가 답하는 질문을 명시
  - 차트는 3~8개 항목, 데이터 라벨 필수
  - Callout은 차트의 So-What을 한 문장으로 요약
  - 우측 패널은 차트가 보여주지 못하는 맥락/해석을 서술
- **deep-briefing 입력**: golden-facts.yaml 수치 → 차트, Evidence → 패널 텍스트
- **논리 흐름**: 질문 → 차트(데이터) → Callout(So-What) → 패널(맥락/해석)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.5 Dual Chart (이중 차트)
- **레이아웃**: 좌 50% 차트A + 우 50% 차트B (범례 공유)
- **필수 요소**: 각 차트 제목(bold), 공통 범례, Source
- **Action Title**: 두 차트의 교차 인사이트 (예: "채택률이 높은 기능에서 생산성 효과도 크다")
- **작성법**: 두 차트가 서로 보완하는 관계여야 함. 독립적 차트 2개 병렬 금지
- **deep-briefing 입력**: 같은 데이터의 다른 절단면 비교
- **논리 흐름**: 질문 → 차트A(측면1) + 차트B(측면2) → 공통 범례 → 교차 인사이트(Callout)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.6 Ranked (#1~N)
- **레이아웃**: 좌 68% (Sub-q + 순위바 + 아이콘 + 산업군) + 우 32% 패널
- **필수 요소**: #1~#5 아이콘(원형, 색상 그라데이션), 라벨, 설명, 산업군 그리드
- **Action Title**: "[대상]은 [항목]을 최우선으로 추진한다"
- **작성법**:
  - 아이콘 색상으로 중요도 표현 (상위 다크초록, 중위 초록, 하위 회색)
  - 각 아이콘 아래 1줄 설명 필수
  - 하단에 산업별 세부 분류 그리드
- **deep-briefing 입력**: Division Findings의 순위/우선순위 데이터
- **논리 흐름**: 질문 → #1~#5 아이콘 순위 → 산업별 세부 그리드 → 패널(맥락/해석)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.7 Process / Flow (프로세스)
- **레이아웃**: 좌 30% 문제 정의 + 우 70% 해결 프로세스
- **필수 요소**: 좌측(제목 + 문제 설명 + 불릿), 우측(방법론 제목 + 최적화 레버 #1/#2 + 쉐브론 스텝)
- **Action Title**: "[방법]으로 [목표]를 확보한다"
- **작성법**:
  - 좌측: 문제/현황을 불릿으로 구조화 (왜 이 프로세스가 필요한지)
  - 우측: 해결 방법론을 #1, #2로 번호화 + 세부 설명
  - 쉐브론 스텝: 3~5단계, 각 단계에 하위 불릿 2~3개
  - 좌측 하단에 KPI 박스 (성과 수치)
- **deep-briefing 입력**: Implementation Playbook → 쉐브론 단계
- **논리 흐름**: 좌측(문제 정의) → 우측(방법론 + 레버 + 쉐브론 스텝) → KPI(성과)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.8 Before/After (현황→AI 전환)
- **레이아웃**: 좌 30% Current + 중 40% AI-first + 우 30% Impact
- **필수 요소**: 좌측(현재 문제점 + 아이콘), 중앙(해결책 제목+설명), 우측(Big Number + 설명)
- **Action Title**: "[주체]는 [전환]으로 [경쟁 우위]를 재구상한다"
- **작성법**:
  - 좌측 문제점은 구체적 수치 포함 (예: "3건 중 2건 지연", "6배 비용 초과")
  - 중앙 해결책은 bold 제목 + 1~2문장 설명
  - 우측 Impact는 Big Number (8~15%, +40% 등) + 단위 설명
- **deep-briefing 입력**: 현황 분석 Findings → 좌측, 전략 제안 → 중앙/우측
- **논리 흐름**: Current(As-Is 문제) → AI-first(To-Be 솔루션) → Impact(정량 효과)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

**subgrid row 보호 규칙**: Before/After 등 subgrid 사용 유형에서, 배경색 컬럼의 accent bar(::before)는 `position: absolute`로 처리하여 grid-row를 차지하지 않아야 한다. accent bar가 grid-row에 포함되면 다른 컬럼의 제목 위치가 밀려남.

**Impact 컬럼 Big Number 배분**: Impact 컬럼의 ba-body에 `justify-content: space-evenly` 적용. `space-between`은 첫/마지막 항목이 컬럼 끝에 밀착되어 불균형하므로, `space-evenly`로 균등 간격 확보.

### J.9 Framework / Diagram (프레임워크)
- **레이아웃**: 전폭 또는 좌 65% + 우 35%
- **필수 요소**: 도식(벤 다이어그램, 매트릭스, 트리 등), 범례 번호(①②③④), 하단 설명
- **Action Title**: "[대상]은 [에코시스템/프레임워크]를 활용하여 [목표]를 달성할 수 있다"
- **작성법**:
  - 도식은 2~4개 영역으로 구분, 각 영역에 번호 부여
  - 하단에 번호별 1줄 설명 그리드
  - 도식 내 텍스트는 10~14px로 작게
  - 주변에 세부 응용 사례 텍스트 배치
- **deep-briefing 입력**: 분석 프레임워크, 기술 분류 체계
- **논리 흐름**: 도식(구조 시각화) → 번호 라벨(①②③④) → 하단 범례(설명) → Action Title이 구조의 핵심 주장
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.10 Table / List (표/목록)
- **레이아웃**: 전폭, 헤더 라벨 + a~f 행 나열
- **필수 요소**: 좌측 라벨(초록 배경 + 알파벳), 라벨 제목, 우측 설명 (bold 핵심 + 상세)
- **Action Title**: "[주제] | [핵심 메시지]" 형태 (라벨 | 구분 허용)
- **작성법**:
  - 각 행은 label + title + description 3요소
  - Description 첫 문장에 bold 핵심 구절
  - 행 수는 4~8개 (3개 미만은 빈약, 9개 이상은 과밀)
  - 행 간 구분선 또는 색상 교대
- **deep-briefing 입력**: 이슈 목록, 과제 정리, 위험 요인 나열
- **논리 흐름**: 행 나열(a→f) → 각 행[현상+원인+필요 행동] → Action Title이 과제 전체 관통 메시지
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.11 Case Study (사례)
- **레이아웃**: 3행 구조 (Impact + Platform + Operating Model 등)
- **필수 요소**: "Illustrative BCG case example" 라벨, Big Number 2~3개, 아이콘, 프로세스 플로우
- **Action Title**: "[사례 맥락] | [방법]으로 [성과]를 달성한다"
- **작성법**:
  - 상단 행: Impact Big Number (80%, 30%+ 등)
  - 중단 행: 방법론/플랫폼 플로우 (화살표 연결)
  - 하단 행: 운영 모델 설명 (불릿)
  - "Illustrative" 라벨로 실제 사례임을 명시
- **deep-briefing 입력**: 경쟁사 사례, 벤치마크 데이터
- **논리 흐름**: Impact(결과 수치) → Platform(방법론 플로우) → Operating Model(실행 체계)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.12 Big Number (핵심 KPI)
- **레이아웃**: 좌 25% KPI 패널 + 우 75% 상세
- **필수 요소**: 대형 숫자(56~80px), 단위, KPI 설명, 우측 세부 데이터
- **Action Title**: "[수치]가 [의미]를 보여준다"
- **deep-briefing 입력**: golden-facts.yaml 핵심 수치
- **논리 흐름**: 핵심 수치(So-What) → 질문(Why-So) → 세부 차트(근거) → Callout(시사점)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.13 Back Cover (뒷표지)
- **레이아웃**: 전폭 다크 배경
- **필수 요소**: 로고, 연락처, copyright, "Read more" 링크
- **작성법**: 텍스트 최소화, 브랜드 일관성
- **논리 흐름**: 없음 — 브랜드 클로징 목적

### J.14 Exec Summary (경영진 요약)
- **레이아웃**: 3행 균등 분할 (라벨 + 본문)
- **필수 요소**: 각 행에 Kicker(영문) + Topic(한글) + 리드 문장 + 불릿 3~5개
- **Action Title**: "Executive summary | [핵심 메시지]"
- **작성법**: 3행이 SCR(Situation→Complication→Resolution)에 대응. 행당 최소 5줄
- **Kicker/Topic 선택 공식**: SCR 스토리라인의 3축에 대응
  - 1행: Kicker = Situation 키워드 (예: "Market", "Context") / Topic = 현황 요약 한글 (예: "거시경제 환경")
  - 2행: Kicker = Complication 키워드 (예: "Cost", "Challenge") / Topic = 문제 요약 한글 (예: "비용 구조 압박")
  - 3행: Kicker = Resolution 키워드 (예: "Growth", "Execution") / Topic = 해결 요약 한글 (예: "성장 전략 실행")
  - Kicker는 영문 대문자, 1단어 권장 (2단어 이내)
  - Topic은 한글 2~4글자, 줄바꿈 허용
- **deep-briefing 입력**: cross-domain-synthesizer 출력
- **논리 흐름**: 3행(S→C→R) 균등 분할 → 각 행[리드+불릿] → Action Title이 전체 종합
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.15 Data-Viz Multi-Column (다중 컬럼 비교)
- **레이아웃**: 전폭, N개 지역/카테고리 반복 컬럼
- **필수 요소**: Sub-question + N개 컬럼(각 컬럼에 차트+데이터 라벨) + 각 컬럼 하단 인사이트 박스
- **Action Title**: "[대상]은 [지역별/세그먼트별] [차이]를 보인다"
- **작성법**: 우측 패널 없음. 각 컬럼 하단에 불릿 3~4개로 해석
- **deep-briefing 입력**: 지역별/세그먼트별 동일 지표 비교 데이터
- **논리 흐름**: 질문 → N개 컬럼 병렬 차트 → 각 컬럼 하단 인사이트 → Action Title이 전체 차이 종합
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.16 Ranked Row-Based (행 반복 순위)
- **레이아웃**: 좌 68% 순위 행 반복 + 우 32% 단일 callout
- **필수 요소**: #1/#2/#3 행(번호+아이콘+제목+설명 1~2줄) + 우측 종합 해석 문단
- **Action Title**: "[대상]이 [최우선 과제]로 유지되고 있다"
- **작성법**: 각 행은 독립적으로 읽을 수 있어야 함. 행간 구분선 필수
- **deep-briefing 입력**: Division Findings 우선순위 데이터
- **논리 흐름**: 질문 → #1/#2/#3 행(아이콘+제목+설명) → 우측 callout(종합 해석)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.17 Divergent Bar (좌우 대칭 막대)
- **레이아웃**: 전폭, N개 행 × 좌우 대칭 수평 막대 + 우측 변화량
- **필수 요소**: Sub-question + 항목 라벨 + 좌(긍정)우(부정) 막대 + 변화량 컬럼 + 그룹 해석 텍스트
- **Action Title**: "[대상]은 [양극화/대조]를 보인다"
- **작성법**: 긍정=초록, 부정=빨강. 국가/세그먼트 그룹 구분. 우측에 해석 블록 2개
- **deep-briefing 입력**: 설문 결과 양극화, 시장별 온도차
- **논리 흐름**: 질문 → 좌우 대칭 막대(긍정/부정) → 변화량 → 그룹별 해석(Annotation)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.18 Split Layout (2톤 배경)
- **레이아웃**: 좌측 다크 배경(핵심 메시지) + 우측 라이트 배경(차트/데이터)
- **필수 요소**: 좌측(큰 폰트 핵심 주장 + Source), 우측(차트 + 데이터 라벨)
- **Action Title**: So-What이 좌측에 큰 폰트로 직접 표시
- **작성법**: 좌측에서 주장(So-What) → 우측에서 데이터 근거(Why-So)
- **deep-briefing 입력**: 핵심 수치 1~2개 + 차트 1개
- **논리 흐름**: 좌측 다크(So-What 직접 표시) → 우측 라이트(차트/데이터 근거)
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.19 Heatmap/Grid (히트맵 격자)
- **레이아웃**: 전폭, N 카테고리 × M 지역/세그먼트 격자
- **필수 요소**: 격자 헤더(M개) + 행 라벨(N개) + 셀 색상 코딩 + 범례 + 그룹 구분
- **Action Title**: "[대상]은 [카테고리별] [지역별] [패턴]을 보인다"
- **작성법**: 초록(증가/긍정)/회색(감소/부정) 색상 코딩. 행 16개 이내
- **deep-briefing 입력**: 다차원 비교 데이터 (카테고리 × 세그먼트)
- **논리 흐름**: N×M 격자(색상 코딩) → 범례 → 그룹 라벨 → Action Title이 전체 패턴 종합
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.20 Scatter Plot (산점도)
- **레이아웃**: 전폭, 2차원 연속형 축
- **필수 요소**: X축/Y축 라벨 + 점/버블 N개(라벨 포함) + 기준선(선택)
- **Action Title**: "[대상]은 [X축] 대비 [Y축]에서 [패턴]을 보인다"
- **작성법**: 강조 포인트는 초록, 약점은 빨강. 라벨이 겹치지 않도록 배치
- **deep-briefing 입력**: 2차원 포지셔닝 데이터 (경쟁 비교, 성과 매핑)
- **논리 흐름**: 2차원 축 배치 → N개 점/버블 → 아웃라이어/클러스터 식별 → Action Title이 포지셔닝 패턴
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.21 Comparison / Dual (좌우 비교)
- **레이아웃**: 좌 50% + 우 50% 대칭 비교
- **필수 요소**: 각 컬럼에 제목 + 본문 2~3문단 + 불릿 3~5개
- **Action Title**: "[A]에서 [B]로의 전환이 [효과]를 가져온다"
- **작성법**: 좌측(As-Is/Before/Traditional) vs 우측(To-Be/After/Digital). 대칭 구조 유지
- **deep-briefing 입력**: Before/After, 옵션 A vs B 비교
- **논리 흐름**: 좌측(As-Is) vs 우측(To-Be) 대칭 → 차이점 구체화 → Action Title이 전환 효과
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### J.22 Matrix / 2×2 (매트릭스)
- **레이아웃**: 전폭, 2차원 격자 (2×2 또는 N×M)
- **필수 요소**: X축/Y축 라벨 + 셀(아이콘+항목명+설명) + 범례
- **Action Title**: "[대상]을 [축1] × [축2] 기준으로 분류한다"
- **작성법**: 각 셀에 아이콘+제목+설명. 전략적 포지셔닝에 사용
- **deep-briefing 입력**: 포지셔닝 분석, 리스크 매트릭스
- **논리 흐름**: 2차원 격자(축1×축2) → 각 셀(아이콘+항목+설명) → Action Title이 분류 패턴의 전략적 의미
- **검증**: 작성 완료 후 D섹션 So-What/Why-So 검증 체크리스트 5항목 실행

### K.1 리서치 결과 유형 → 슬라이드 유형 선택

```
리서치 결과를 받았다. 어떤 장표를 만들 것인가?

1. 핵심 주장 1개를 전달하고 싶다
   ├─ 숫자가 핵심이다
   │  ├─ 수치 1~2개 + 우측 세부 차트 → Big Number (J.12)
   │  └─ 수치 + 좌측 대형 텍스트 강조 → Split Layout (J.18)
   ├─ 데이터 근거 + 해석 패널
   │  ├─ 단일 차트 + 우측 해석 → Insight+Chart (J.4)
   │  └─ 같은 데이터의 2개 절단면 → Dual Chart (J.5)
   ├─ 지역/세그먼트별 비교 → Data-Viz Multi-Column (J.15)
   └─ 논리적 구조로 설명 → Text-heavy / Introduction (J.3)

2. 여러 항목을 비교하고 싶다
   ├─ 순위가 있다
   │  ├─ 항목 3~5개 + 각 항목에 설명 필요 → Ranked Row-Based (J.16)
   │  └─ 항목 5개 + 아이콘 시각화 + 산업별 세부 → Ranked Icon Grid (J.6)
   ├─ 전/후 변화
   │  ├─ 3컬럼 (현황→전환→효과) → Before/After (J.8)
   │  └─ 2컬럼 대칭 비교 → Comparison (J.21)
   ├─ 2차원 축으로 포지셔닝
   │  ├─ 범주형 축 (2×2 또는 N×M) → Matrix (J.22)
   │  └─ 연속형 축 (상관관계) → Scatter Plot (J.20)
   ├─ 좌우 대칭 (긍정/부정) → Divergent Bar (J.17)
   └─ 다차원 격자 (카테고리 × 세그먼트) → Heatmap/Grid (J.19)

3. 프로세스/방법론을 설명하고 싶다
   ├─ 문제 정의 + 해결 단계 → Process (J.7)
   └─ 개념 구조/분류 체계/생태계 → Framework (J.9)

4. 문제/과제를 나열하고 싶다
   ├─ 항목이 4~8개다 → Table/List (J.10)
   └─ 사례로 증명해야 한다 → Case Study (J.11)

5. 데이터를 시각화하고 싶다
   ├─ 같은 데이터의 다른 절단면 비교 → Dual Chart (J.5)
   ├─ 2차원 연속형 축 → Scatter Plot (J.20)
   └─ 2차원 카테고리 격자 → Heatmap/Grid (J.19)

6. 경영진 요약이 필요하다 → Exec Summary (J.14)

7. 섹션을 구분해야 한다 → Agenda (J.2)
```

### K.4+ 장 간 전환 규칙

앞 장의 So-What이 뒷 장의 배경이 되는 **체인 구조**:

| 전환 | 앞 장 유형 → 뒷 장 유형 | 연결 논리 | 연결 문장 패턴 |
|------|---------------------|---------|-------------|
| S→S | Insight+Chart → Insight+Chart | 동일 주제 심화 | "이러한 [트렌드]는 [세부 영역]에서 더 두드러진다" |
| S→C | Insight+Chart → Big Number | 트렌드 → 핵심 문제 수치 | "[현황] 속에서 [문제 수치]가 주목된다" |
| C→C | Big Number → Ranked/Table | 문제 규모 → 세부 요인 | "[문제]의 핵심 동인은 다음과 같다" |
| C→R | Table/Ranked → Process | 문제 요인 → 해결 방법론 | "이러한 과제를 해결하기 위해 [방법론]을 적용한다" |
| R→R | Process → Before/After | 방법론 → 구체적 전환 | "[방법론]을 적용하면 [As-Is]에서 [To-Be]로 전환된다" |
| R→R | Before/After → Case Study | 전환 모델 → 실증 사례 | "실제로 [기업]은 이 접근법으로 [성과]를 달성했다" |

**핵심 원칙**:
1. 각 장의 Action Title은 이전 장의 So-What에서 자연스럽게 이어져야 한다
2. 독자가 "왜 이 장이 여기에 있는가?"를 의문 없이 이해할 수 있어야 한다
3. Agenda 슬라이드는 대주제 전환(S→C, C→R) 시에만 삽입

### K.5 장표 구성 시 핵심 원칙

1. **1슬라이드 = 1메시지**: 2개 이상의 주장을 한 장에 넣지 않는다
2. **데이터가 있으면 차트로**: 텍스트로 숫자를 나열하지 말고 시각화한다
3. **So-What 필수**: 차트 아래에 반드시 Callout으로 시사점을 명시한다
4. **빈 공간 = 콘텐츠 부족**: 영역이 남으면 Evidence를 추가하거나 레이아웃을 변경한다
5. **Action Title이 결론**: 제목만 읽어도 슬라이드의 핵심 주장을 이해할 수 있어야 한다
6. **패널은 맥락 제공**: 우측 패널은 차트가 보여주지 못하는 해석/배경을 서술한다
7. **전환이 자연스러워야 한다**: 장 간 논리적 연결 없이 유형만 바꾸지 않는다
8. **Agenda는 구조적 전환점**: S→C, C→R 대전환 시에만 사용 (소주제 전환에는 불필요)
9. **Exec Summary는 예고편**: 본문을 읽기 전 S/C/R 각 축의 핵심 메시지를 미리 전달
10. **Case Study는 증거**: Resolution의 실현 가능성을 구체적 사례로 증명

### K.6 에이전트 실행 워크플로우

```
Step 1: Division Findings → 슬라이드 유형 결정 (K.1 의사결정 트리)
Step 2: Action Title 작성 (J 유형별 공식 + D 마스터 테이블)
Step 3: section-map에서 해당 유형의 섹션 구조 로드
Step 4: golden-facts.yaml에서 수치 매핑 → 차트/Big Number 배치
Step 5: Evidence → 패널/불릿 텍스트 작성 (D 패널 작성법)
Step 6: CSS 클래스 적용 (slide-system.css의 layout-* 클래스)
Step 7: So-What/Why-So 검증 체크리스트 5항목 실행
Step 8: verify-slide-layout.js로 렌더링 검증
Step 9: 스크린샷 → BCG 레퍼런스 비교 → 빈 공간 10% 미만 확인
```

## L. 강제 규칙 (Mandatory Spec)

> 이 섹션의 규칙은 **예외 없이 강제**된다. 위반 시 장표가 아닌 **리서치 콘텐츠를 보강**한다.

### L.1 핵심 설계 원칙
- 리서치 내용은 **방대하다는 전제**로 설계한다
- 장표를 채우기에 내용이 부족하면 **리서치를 되돌아가서 보강**한다
- 폰트를 줄이거나, 여백을 늘리거나, 장표를 축소하는 것은 **금지**한다

### L.2 폰트 크기 고정값 (Canonical Spec — 유일한 정본)

> 이 표가 폰트 크기의 **유일한 정본**이다. section-map.md, CSS 변수 모두 이 표를 따른다.
> B 섹션의 통계 표는 "PDF 원본 참고값"이며, 이 표가 최종 확정값.

| 역할 | CSS 변수 | 고정값 | 실측 근거 (PDF pt×2) |
|------|---------|--------|---------------------|
| Action Title | `--fs-title` | 48px | 24pt×2=48px (Codex 4회차 확인) |
| Sub-question | `--fs-subq` | 32px | 16pt×2=32px (실측 31.9px) |
| 컬럼/섹션 제목 | `--fs-col-title` | 28px | 14pt×2=28px |
| 패널 본문 (≤3문단) | `--fs-panel` | 32px | 15~16pt×2=30~32px. CSS: `.panel-text-short` 사용. 4문단 이상은 `--fs-body-lg` |
| 본문 강조 | `--fs-body-lg` | 24px | 12pt×2=24px (실측 23~24px) |
| 본문 일반 | `--fs-body` | 24px | 11.5~12pt×2=23~24px |
| Callout | `--fs-callout` | 24px | 12pt×2=24px |
| Source/각주 | `--fs-source` | 20px | 10pt×2=20px (실측 19.9px) |
| 차트 값 | `--fs-chart-val` | 28px | 14pt×2=28px |
| 페이지 번호 | `--fs-page` | 24px | 12pt×2=24px (실측 24px) |
| 순위 번호 | `--fs-rank` | 56px | 28pt×2=56px (실측 55.9~64px) |
| Big Number | `--fs-big` | 72px | 36pt×2=72px (실측 72~80px) |
| 인라인 KPI | `--fs-kpi-inline` | 36px | 18pt×2=36px |

> **한글 보정**: 위 값은 Henderson Sans 기준. Pretendard 한글은 85~90% 축소 적용하여 CSS `:lang(ko)`에서 오버라이드.

**규칙**:
- 위 CSS 변수 외의 `font-size` 인라인 값 사용 **금지**
- 모든 텍스트는 `var(--fs-*)` 형태로만 지정
- 예외 없음 — Agenda, Cover 포함
- **본문 최소 크기 원칙**: slot-body 내 독자가 읽어야 하는 모든 텍스트는 **최소 `--fs-body` (한글 22px)** 이상
- `--fs-source` (한글 18px)는 Source/각주/보조 라벨에만 허용
- `--fs-kicker` (14px)는 Exec Summary kicker에만 허용 (L.2 정본 외 예외)

### L.3 콘텐츠 밀도 요구량 (강제)

| 영역 | 최소 요구량 |
|------|-----------|
| Table/List 각 행 | bold 핵심문 1개 + 상세 설명 1~2문장 (최소 2줄) |
| Before/After 각 항목 | bold 제목 + 설명 1줄 이상 |
| Case Study 각 섹션 | 3개 이상 콘텐츠 요소 (수치+설명 또는 플로우+설명) |
| 우측 패널 (secondary-panel) | 최소 3문단, 각 문단에 bold 핵심구 1개 |
| 차트 영역 | `flex:1`로 가용 공간 채움, 차트 항목 최소 4개 |
| Exec Summary 각 행 | 리드 문장 1개 + 불릿 3개 + 마무리 문장 1개 |
| 좌측 패널 (col-left-narrow) | 제목 + 설명 + 불릿 5개 + KPI 박스 |
| Insight+Chart 차트 영역 | 차트 항목 최소 4개, 데이터 라벨 필수, 범례 포함 |
| Ranked (Icon Grid) | 아이콘 5개, 각 아이콘 아래 라벨+설명, 하단 산업군 그리드 |
| Ranked (Row-Based) | 행 3~5개, 각 행에 순위번호+아이콘+제목+설명 1~2줄, 우측 callout |

**콘텐츠가 부족하면**: 폰트 축소 ❌ → 리서치 보강 ✅

### L.4 인라인 스타일 규칙

```
허용:
- layout 관련: display, gap, padding, margin, flex, grid
- 색상: color, background (CSS 변수 사용 권장)
- 크기: width, height, min-width

금지:
- font-size: [숫자]px → 반드시 var(--fs-*) 사용
- font-size: var(--fs-source) → 본문 텍스트에 사용 금지 (Source/각주 전용)
- font-weight 단독 지정 → CSS 클래스 사용 권장
- gap/margin/padding에 px 직접 지정 금지 → 반드시 `var(--sp-*)` 토큰 사용
- 행 반복 레이아웃 container에 padding-left/right + background 동시 사용 금지 (P6 위반)
```

## M. 장표 생성 시 필수 규칙 (에이전트용)

> 이 섹션은 deep-briefing의 report-writer 및 슬라이드 생성 에이전트가 반드시 따라야 하는 규칙이다.

### M.1 장표 생성 절차

```
1. 리서치 결과(findings, golden-facts)에서 슬라이드 유형 결정 (K.1 의사결정 트리)
2. 해당 유형의 HTML 템플릿 적용 (J 섹션 참조)
3. CSS 변수로만 폰트 크기 지정 (L.2 고정값 표)
4. 각 영역에 콘텐츠 밀도 요구량 충족 (L.3)
5. verify-slide-layout.js 실행 → overflow 0건 + font-check PASS
6. 스크린샷으로 빈 공간 없음 확인
```

### M.2 금지 사항

1. `font-size: [숫자]px` 인라인 사용 금지 → `var(--fs-*)` 사용
2. 섹션에 1줄만 넣기 금지 → 최소 2줄
3. 빈 공간을 여백으로 남기기 금지 → 콘텐츠 추가
4. 콘텐츠 부족 시 폰트 축소 금지 → 리서치 보강
5. CSS 클래스 없이 레이아웃 구성 금지 → `layout-*` 클래스 사용

### M.3 콘텐츠 부족 시 처리 흐름

```
빈 공간 발견
  ↓
콘텐츠를 추가할 수 있는가?
  ├─ Yes → Evidence/Data 추가 → 재검증
  └─ No → 리서치 보강 요청 (report-writer → Division Lead 재실행)
         → 절대로 폰트 축소, 여백 확대, 레이아웃 축소 하지 않음
```
