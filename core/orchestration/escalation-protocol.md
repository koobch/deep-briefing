# 에스컬레이션 & 되돌아가기 프로토콜

> 이상 상황 발생 시 보고 체인, 심각도 분류, 되돌아가기(Backtrack) 규칙.
> 모든 에이전트가 이 프로토콜을 참조한다.

## 설계 원칙

1. **문제는 빨리 올라가고, 해결은 가능한 낮은 레벨에서** — 보고는 즉시, 처리는 최소 범위
2. **심각도에 비례한 대응** — 사소한 문제로 전체를 멈추지 않는다
3. **되돌아가기는 비용** — 필요할 때 하되, 가볍게 결정하지 않는다
4. **모든 이상 상황은 기록** — 재발 방지를 위해 로그 필수

---

## 에스컬레이션 체인

```
Leaf → Lead (또는 Sub-lead) → Division Lead → PM → 사용자
```

각 레벨에서 처리 가능하면 상위로 올리지 않는다.
처리 불가 시에만 상위로 에스컬레이션한다.

---

## 이상 상황 분류

### Type 1: 데이터 이슈

| 상황 | 심각도 | 처리 주체 | 대응 |
|------|--------|----------|------|
| 소스 1개 접근 불가 | 경미 | Leaf | 대체 소스 탐색, 자율 처리 |
| 소스 2개+ 접근 불가 | 중간 | Lead | 다른 리프에 교차 확인 요청 |
| 핵심 데이터 수집 불가 | 중대 | PM | data_gap 명시 + 사용자에 데이터 요청 |
| 사용자 제공 데이터 품질 불량 | 중대 | PM → 사용자 | 데이터 재요청 또는 분석 범위 조정 |

### Type 2: 가설/분석 이슈

| 상황 | 심각도 | 처리 주체 | 대응 |
|------|--------|----------|------|
| 보조 가설 반증됨 | 경미 | Leaf | 가설 수정, 자율 반복 |
| 핵심 가설 반증됨 | 중대 | Lead → PM | Claim rejected 기록 + 대안 가설 제시 |
| Division 전제 붕괴 | 치명 | PM → 사용자 | 리서치 방향 재설정 필요 |
| 리프 간 해소 불가 모순 | 중간 | Lead → PM | key_tensions 기록 + Sync Round에서 처리 |

### Type 3: Division 간 이슈

| 상황 | 심각도 | 처리 주체 | 대응 |
|------|--------|----------|------|
| cross_domain 질문 응답 불가 | 중간 | PM | 질문 재구성 또는 범위 축소 |
| Division 간 Claim 모순 | 중대 | fact-verifier → PM | 교차 검증 후 해소 또는 사용자 판단 |
| 전략 핵심 전제 검증 실패 | 치명 | PM → 사용자 | 전략 방향 재설정 |

### Type 4: 시스템 이슈

| 상황 | 심각도 | 처리 주체 | 대응 |
|------|--------|----------|------|
| 리프 에이전트 비정상 종료 | 경미 | Lead | 1회 재스폰 |
| 재스폰 후 재실패 | 중간 | Lead → PM | data_gap 처리 또는 다른 리프에 범위 재배분 |
| API 쿼터 소진 | 중간 | PM | 남은 쿼터 재배분 또는 대체 소스 |
| 전체 Division 실패 | 치명 | PM → 사용자 | 해당 Division 분석 없이 진행 가능한지 판단 |

## 심각도별 처리 규칙

### 경미 (Minor)
```
처리: 해당 에이전트가 자율 처리
보고: iteration_log에 기록 (상위 보고 불필요)
시간: 자율 반복 내에서 해결
영향: 개별 Claim의 confidence 조정 수준
```

### 중간 (Moderate)
```
처리: Lead가 조정 (리프 간 재배분, 교차 확인 등)
보고: Lead 출력의 contradictions_resolved 또는 data_gaps에 기록
시간: Division 내에서 해결, Sync Round 전까지
영향: Division 일부 Claim에 영향, 전체 방향은 유지
```

### 중대 (Major)
```
처리: PM이 판단 — 사용자 보고 여부 결정
보고: PM이 Sync Briefing에 포함 또는 별도 에스컬레이션
시간: Sync Round에서 처리, 필요 시 추가 Phase
영향: 전략 방향 일부 수정 가능
```

### 치명 (Critical)
```
처리: 즉시 중단 — PM이 사용자에게 보고
보고: 즉시 (Sync Round까지 기다리지 않음)
시간: 사용자 판단 후 재개
영향: 전략 방향 전면 재설정 가능

에스컬레이션 형식:
┌─────────────────────────────────────────────┐
│ 🚨 Critical Escalation                       │
│                                               │
│ 상황: [구체적 상황 설명]                       │
│ 영향: [이것이 전략에 미치는 영향]              │
│ 원인: [왜 이런 상황이 발생했는지]              │
│                                               │
│ 선택지:                                       │
│ A. [방향 전환 옵션] — trade-off 설명           │
│ B. [축소 진행 옵션] — trade-off 설명           │
│ C. [중단 옵션] — 사유                          │
│                                               │
│ 추천: [추천안 + 근거]                          │
└─────────────────────────────────────────────┘
```

## 되돌아가기 (Backtrack) 프로토콜

### 모드별 가용성

| 모드 | Division 내 (리프 재작업) | Phase 되돌아가기 | 전면 되돌아가기 |
|------|------------------------|----------------|--------------|
| Auto | Lead 자율 | 불가 | 불가 |
| Interactive | Lead 자율 | 사용자 트리거만 | 사용자 트리거만 |
| Team | Lead → PM 보고 | PM 제안 + 사용자 승인 | PM 제안 + 사용자 승인 |

※ Division 내 리프 재작업은 "되돌아가기"가 아닌 **품질 보증의 일환**이므로 모든 모드에서 허용.

### 되돌아가기 판단 기준

```
"되돌아가기가 필요한가?"를 판단하는 의사결정 트리:

1. 이슈가 현재 Phase 내에서 해결 가능한가?
   → Yes: 되돌아가기 불필요, 현재 Phase에서 수정
   → No: 다음 질문

2. 이슈가 이전 Phase의 산출물 오류에서 기인하는가?
   → Yes: 되돌아가기 후보
   → No: 현재 Phase의 문제 — 에스컬레이션으로 처리

3. 되돌아가기의 비용 vs 보정의 비용?
   → 되돌아가기 비용 < 보정 비용: 되돌아가기
   → 되돌아가기 비용 > 보정 비용: 현재 위치에서 보정 + 한계 명시
   (비용 = 재작업할 에이전트 수 × 재작업 범위)

4. 되돌아가기 범위는?
   → 특정 Division만: Division 내 재작업 (Lead 관할)
   → 전체 Division: PM 판단 + 사용자 확인 (Team 모드)
```

### 되돌아가기 유형

#### A. Division 내 되돌아가기 (Lead 관할)

```
상황: 리프 A의 출력이 리프 B에 의해 반증됨
범위: 리프 A만 재작업
처리: Lead가 리프 A에 수정 지시 → 재수집 → 재합성
승인: Lead 자율 판단 (PM 보고만)

예시:
leaf-agent-A: "카테고리B CAGR 15%"
leaf-agent-B: "주요 지역별 합산 결과 12%에 가까움"
→ leaf-agent-A에 재확인 지시
```

#### B. Phase 되돌아가기 (PM 관할)

```
상황: Phase 2(심화 리서치) 결과가 Phase 1(초기 리서치)의 전제를 부정
범위: 해당 Division의 Phase 1부터 재실행
처리:
  1. PM이 영향 범위 분석
  2. 영향받는 Division 식별
  3. 사용자 확인 (Interactive/Team)
  4. 해당 Division만 Phase 1 재실행
  5. 다른 Division은 유지 (영향 없으면)
승인: PM + 사용자 확인 (Team 모드)

예시:
Phase 2에서 capability-lead가 "자체 개발 불가" 판정
→ Phase 1의 product-lead 전제 ("자체 개발 기준 설계") 붕괴
→ product Division Phase 1부터 재실행 (공동개발 기준으로)
```

#### C. 전면 되돌아가기 (사용자 관할)

```
상황: Client Brief의 핵심 전제가 틀렸거나 변경됨
범위: Phase 0부터 재실행
처리:
  1. PM이 치명 에스컬레이션
  2. 사용자와 Client Brief 재확인
  3. Research Plan 재수립
  4. 전체 Division 재실행
승인: 사용자 명시적 승인 필수

예시:
사용자: "아, 2028년이 아니라 2027년 초 론칭이 목표였어"
→ 타임라인 변경으로 전체 전략 재설계 필요
```

### 되돌아가기 비용 계산

```
되돌아가기 비용 = 재작업 에이전트 수 × 재작업 깊이

깊이 1: 리프 1개 재실행 (경미)
깊이 2: Sub-lead + 하위 리프 재실행 (중간)
깊이 3: Division Lead + 전체 리프 재실행 (중대)
깊이 4: 여러 Division 재실행 (치명)

원칙: 최소 범위로 되돌아간다.
"leaf-agent-A 하나만 재실행하면 되는데 Market Division 전체를 재실행"하지 않는다.
```

## 이력 관리

### 에스컬레이션 로그

모든 중간 이상 이상 상황은 로그에 기록한다.

```yaml
# {project}/escalation-log.yaml

escalations:
  - id: ESC-001
    timestamp: YYYY-MM-DD
    severity: moderate | major | critical
    type: data | hypothesis | cross-division | system
    reporter: {agent-id}
    description: "상황 설명"
    resolution: "처리 결과"
    impact: "전략/분석에 미친 영향"

backtracks:
  - id: BT-001
    timestamp: YYYY-MM-DD
    type: division-internal | phase | full
    trigger: auto | lead | user
    from_phase: {N}
    to_phase: {N}
    affected_divisions: [division-ids]
    reason: "되돌아가기 사유"
    cost: "재작업 에이전트 수 × 깊이"
    outcome: "결과"
```

### 자가발전 연계

리서치 완료 후 PM이 에스컬레이션 로그를 분석:
- 반복 패턴 식별 → EP(Error Pattern)로 등록
- 특정 에이전트의 반복적 이슈 → agent.md 개선
- 특정 데이터 소스의 반복적 실패 → 소스 접근성 로그 업데이트

## 모드별 의사결정 권한 요약

| 결정 | Auto | Interactive | Team |
|------|------|------------|------|
| 리프 재작업 (품질 보증) | Lead 자율 | Lead 자율 | Lead 자율 |
| Division 내 되돌아가기 (리프 재작업) | Lead 자율 | Lead 자율 | Lead → PM 보고 |
| Phase 되돌아가기 | 불가 | 사용자 트리거만 | PM 제안 + 사용자 승인 |
| 전면 되돌아가기 | 불가 | 사용자 트리거만 | PM 제안 + 사용자 승인 |
| 데이터 추가 요청 | PM 자동 | PM → 사용자 확인 | PM → 사용자 확인 |
| 분석 범위 축소 | PM 자율 | PM → 사용자 확인 | PM 제안 + 사용자 승인 |
| 리서치 중단 | 불가 | 사용자만 | PM 제안 + 사용자 승인 |

※ "리프 재작업"과 "Division 내 되돌아가기"의 차이:
- 리프 재작업: 반려 조건 미충족 → 재작업 지시 (품질 보증, 모든 모드에서 자율)
- Division 내 되돌아가기: 리프 A의 결론이 리프 B에 의해 반증 → 가설 수정 후 재리서치
