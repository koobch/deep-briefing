01

프로젝트 개요

Deep-Briefing이란

Deep-Briefing은 49개의 전문 AI 에이전트가 협력하여 수행하는 전략 리서치 시스템입니다. 4계층 에이전트 계층(PM → Lead → Leaf + Cross-cutting/QA)이 병렬로 작동하며, 파일 시스템 기반 통신으로 프로젝트 의존성 없이 자동 스케일합니다.

핵심 특징
  • Division 병렬 리서치 (핵심 4 + 확장 최대 3)
  • 결정 가능한 분석 (Many Know, Few Decide → We Decide)
  • 4단계 검증 체계 (VL-1~3 + QA)
  • 자동 학습 엔진 (Layer 0~3 지식 시스템)
  • 토큰 최적화 (94% 절감, compile-lead-context.sh)
  • 3-모드 체계 (Auto / Interactive / Team)
02

딥리서치 차별점

차별점 기존 리서치 Deep-Briefing
1. 결정성 많은 정보 제공 ("이런 게 있어요") 명확한 판단 ("이걸 해/하지 마")
2. 수치 검증 단순 인용 출처 추적 + Golden Facts 교차 검증 (VL-3)
3. 반증 포함 확증 편향 (우호적 정보만) Red Team이 핵심 전제 반증 (Phase 3)
4. 반복 심화 한 번에 완성 Phase 1→Sync→Phase 2→Sync 반복 심화
5. 맥락 맞춤 산업 일반론 Client Brief → Division Brief 맞춤 분석
6. 다차원 분석 단순 관점 Market/Product/Capability/Finance 동시 분석 + 교차

04

에이전트 상세

Deep-Briefing은 총 49개의 전문 에이전트로 구성됩니다. 각 에이전트는 명확한 역할 경계와 검증 책임을 가지며, 4계층 구조로 조직되어 병렬 리서치를 수행합니다.

4.1 에이전트 계층 구조 (4-Layer)

계층 구조
Layer 역할 에이전트 수 책임
Layer 0 PM research-pm 1 전체 오케스트레이션, Phase 0~5.5 관리, Division Lead 스폰
Layer 1 Division Lead 핵심 4 + 확장 3 7 각 Division의 분석 총괄, Leaf 스폰, VL-1.5 삼각 검증
Layer 2 Leaf 분석가 Division별 MECE 전문가 26 MECE 분석 실행, VL-1 자가 검증 (2소스+, 반증 검토)
Layer 3 Cross-cutting 검증 + 합성 + 보고 + QA + 유틸 14 Division 횡단 검증, 통합, 보고서 생성, 품질 보증

최대 네스팅 깊이는 Level 2입니다: PM → Division Lead → Leaf. Cross-cutting 에이전트는 PM이 직접 스폰하므로 별도 네스팅이 발생하지 않습니다.

4.2 PM 에이전트 (research-pm)

역할

리서치 전체를 오케스트레이션하는 최상위 에이전트. 클라이언트 브리프 수집부터 최종 보고서 납품까지 Phase 0~5.5의 모든 단계를 관리하며, Division Lead를 스폰하고 Sync Round를 조율합니다.

스폰 대상
  • 핵심 Division Lead 4개 (Market, Product, Capability, Finance) — 거의 모든 리서치에서 활성화
  • 확장 Division Lead 0~3개 (People-Org, Operations, Regulatory) — 주제에 따라 선택 투입
  • Cross-cutting 에이전트 — Phase 3 이후 검증/합성/보고/QA 에이전트를 단계별 스폰
실행 모드 (3-모드 체계)
모드 특징 적합 상황
Auto 직행. 가설 자동 확정, 피드백 생략 반복 리서치, 빠른 탐색이 필요한 경우
Interactive 매 단계 사용자 게이트 + 가설 검토 + 피드백 중요 의사결정, 방향성 확인이 필요한 경우
Team Deep Discovery + 토론 + 전방위 되돌아가기 복잡한 전략 과제, 다각도 검토가 필요한 경우
PM 의사결정 포인트
Phase 의사결정 설명
Phase 0 Division 선택 클라이언트 브리프 분석 후 활성화할 Division 결정 (핵심 4 + 확장 0~3)
Phase 0.5 가설 확정 Quick Scan 결과로 3~5개 가설 도출 → 사용자 정렬 후 Division Briefs에 반영
Sync R1 방향 조정 Phase 1 결과 기반 데이터 갭 식별, Phase 2 심화 방향 설정
Phase 3 사고 루프 관리 검증/도전/반론 에이전트 스폰, 수렴 판정
Phase 4 보고서 승인 SCR 스토리라인 구조 검토, 최종 보고서 승인

4.3 Division Lead (7개)

Division Lead는 각 분석 영역의 총괄 책임자입니다. PM으로부터 Division Brief를 받아 Leaf 에이전트를 스폰하고, 결과를 합성하며, VL-1.5(삼각 검증) 및 VL-2(정합성) 검증을 수행합니다.

Lead 스폰 방식
  • 모드 A (tmux 병렬, 기본): spawn-leads.sh로 tmux 독립 CLI 스폰. 완전 병렬 실행. 가장 빠른 처리
  • 모드 B (Agent tool, tmux 불가 시): PM CLI 내에서 Agent tool로 Lead 스폰. 2개씩 순차-병렬 실행
Division Lead 약자 구분 분석 범위 Leaf 수 활성화 기준
market-lead M 핵심 시장 규모, 고객, 경쟁, 채널, 역학 5 거의 항상
product-lead P 핵심 제품, 차별화, GTM, 가격 4 거의 항상
capability-lead C 핵심 기술, 인재, 자산, 실행력 4 거의 항상
finance-lead F 핵심 매출, 비용, 투자, 가치평가 4 거의 항상
people-org-lead H 확장 조직설계, 인재전략, 문화 3 조직/인력/문화 이슈 시
operations-lead O 확장 프로세스, 공급망, 인프라 3 운영/인프라 이슈 시
regulatory-lead R 확장 규제준수, 규제전망, ESG 3 규제/법률/ESG 이슈 시
Division Lead 공통 책임
  • Leaf 스폰 & 관리: Division Brief 기반으로 Leaf 에이전트 스폰, 분석 결과 수집
  • VL-1.5 삼각 검증: Leaf 결과물에 대한 삼각 검증 + 스팟체크 수행
  • VL-2 정합성 검증: 엔터티/시점/정의 일관성 확인
  • 합성 보고: Leaf 분석 결과를 Division 수준으로 합성하여 findings/에 출력
  • 학습 엔진: Phase 1/2 완료 후 데이터 소스 신뢰도, 프레임워크 효과성, 분석 패턴 등을 learnings/에 추출

4.4 Leaf 분석가 (26개)

Leaf 에이전트는 Division Lead가 스폰하는 최하위 분석 단위입니다. 각 Leaf는 MECE 기반의 분석 프레임워크를 내장하고 있으며, Phase 1(초기 리서치)과 Phase 2(심화 리서치)에서 활성화됩니다. 모든 Leaf는 VL-1(자가 검증: 2소스 이상 + 반증 검토)을 수행합니다.

Market Division (5)

Leaf 에이전트분석 범위MECE 분석 프레임
market-sizing 시장 규모, 성장률, 예측, 세분화 TAM/SAM/SOM 산출, CAGR 추정, 시장 세분화 매트릭스 (지역/제품/고객별), Top-Down vs Bottom-Up 교차 검증
customer-analysis 고객 세그먼트, 니즈, 의사결정 구조 페르소나 정의, 구매 여정(Awareness~Advocacy) 매핑, 전환 퍼널 분석, 이탈(Churn) 원인 분석, 세그먼트별 가치 평가
competitive-landscape 경쟁사 지형, 점유율, 전략 Porter 5 Forces 분석, 경쟁 포지셔닝 맵, 점유율 워터폴(증감 요인), 경쟁사별 강점/약점 매트릭스
channel-landscape 유통 채널 구조, 파트너 유통 구조(직접/간접/하이브리드) 분해, 채널별 마진 스택, 채널 파트너 역할 분석, 채널 갈등(Conflict) 식별
market-dynamics 매크로 트렌드, 규제, 기술 영향 PESTLE 분석, 기술 S-커브 단계 판정, 규제 변화 영향도 매핑, 산업 전환점(Inflection Point) 식별

Product Division (4)

Leaf 에이전트분석 범위MECE 분석 프레임
product-offering 제품 정의, 포트폴리오, 로드맵 기능 매트릭스(Must/Should/Could/Won't), Kano 모델(기본/성능/감동), 제품 로드맵 분석, 기술 부채 평가
value-differentiation 가치 명제, 차별화, PMF PMF(Product-Market Fit) 검증, JTBD(Jobs To Be Done) 프레임워크, 경쟁 우위 지속가능성 분석, 가치 명제 캔버스
go-to-market 유통 전략, 영업 모델 채널 전략(직접/파트너/마켓플레이스), 영업 모델(PLG/SLG/CLG) 비교, 파트너십 구조 분석, 지역별 GTM 차이
pricing-monetization 가격 전략, 수익 모델 가격 탄력성(Price Elasticity) 추정, 번들링/언번들링 전략, LTV/CAC 분석, 수익 모델 비교(구독/일시불/프리미엄)

Capability Division (4)

Leaf 에이전트분석 범위MECE 분석 프레임
technology-ip R&D 파이프라인, 특허, 기술 성숙도 R&D 투자 대비 산출물 효율, 특허 포트폴리오(인용/피인용), 기술 성숙도(TRL) 평가, 기술 스택 현대화 수준

strategic-assets 브랜드, 데이터, 생태계 네트워크 브랜드 자산 가치 추정, 데이터 자산(규모/유일성/활용도), 생태계 네트워크 효과 분석, 파트너십 가치 매핑
execution-readiness 실행 트랙레코드, 민첩성, 변화 관리 과거 전략 실행 성공률, 조직 민첩성(의사결정 속도/적응력), 변화 관리 역량(Change Readiness), 실행 병목 식별

Finance Division (4)

Leaf 에이전트분석 범위MECE 분석 프레임
revenue-growth 매출 드라이버, 전망 시나리오 매출 드라이버 분해(가격 x 수량 x 믹스), 코호트 분석(신규/기존/이탈), 시나리오별 전망(Bull/Base/Bear), 매출원 다각화
cost-efficiency 비용 구조, 벤치마킹 비용 구조 분해(고정/변동, COGS/OPEX), 동종 업계 벤치마킹, 단위 경제학(Unit Economics), 스케일 효과 추정
investment-returns ROIC, IRR, DCF, 투자 우선순위 ROIC/WACC 스프레드, IRR 시뮬레이션, 현금흐름 DCF 모델링, 투자 우선순위 매트릭스(영향 x 실현 가능성)
valuation-risk 멀티플, 밸류에이션, 리스크 멀티플 분석(EV/EBITDA, P/E, P/S), 시나리오별 밸류에이션, 리스크 매트릭스(확률 x 영향), 민감도 분석(토네이도 차트)

process-excellence 프로세스, 자동화, 품질 린/식스시그마 성숙도, 프로세스 마이닝(병목/재작업), 자동화 기회 매트릭스(효과 x 난이도), 품질 비용(CoQ) 분석
supply-chain 공급망 네트워크, 리드타임, 리스크 공급망 네트워크 맵(Tier 1/2/3), 리드타임 분해, 리스크 매핑(지정학/기후/의존도), 재고 최적화(EOQ/Safety Stock)
infrastructure IT/물리 인프라, 확장성, 기술 부채 IT 인프라 현대화 수준, 확장성 한계(현재 용량 vs 예상 수요), 기술 부채 정량화, 보안/복구력(RTO/RPO) 평가

4.5 Cross-cutting 에이전트 (15개)

Cross-cutting 에이전트는 Division 경계를 넘어 작동합니다. 검증, 합성, 보고, QA의 4가지 기능 그룹으로 나뉘며, PM이 Phase별로 직접 스폰합니다.

검증 에이전트 (Phase 3) — 4개

fact-verifier — 수치 진실성 파수꾼
  • 역할: VL-3 교차 검증 수행. Division 간 수치 정합성을 확인하고, 불일치 시 원본 소스까지 추적
  • Golden Facts 관리: findings/golden-facts.yaml의 유일한 수정 권한 보유. 모든 핵심 수치의 SSOT(Single Source of Truth)
  • 검증 방법: 2개 이상 독립 소스 교차, 시점 일치 확인, 단위/정의 통일, 이상치 플래그
  • 활성 시점: Sync Round 1/2, Phase 3
logic-prober — 수직 논리 검증
  • 역할: 재귀적 Why Chain(5 Whys)으로 인과 관계를 수직 추적. 논리 비약(Logical Leap)을 탐지
  • 검증 대상: 각 Division의 핵심 Claim → Evidence → Data → Source 체인의 논리적 타당성
  • 산출물: 논리 비약 목록, 전제 검증 결과, 추가 검증 필요 항목 식별
  • 활성 시점: Phase 3
strategic-challenger — 수평 5-레인 도전
  • 역할: 5개 레인에서 동시 수평 도전을 수행하여 분석의 사각지대를 노출
  • 5-레인 구성:
    • 대안 시장: 간과된 시장 기회나 위협
    • 기술 변곡: 기술 변화가 분석 전제를 뒤집을 가능성
    • 규제 리스크: 규제 변화로 인한 전략 무효화 시나리오
    • 조직 관성: 실행 가능성을 저해하는 조직적 장벽
    • 타이밍: 시간 변수(너무 빠름/느림)가 전략에 미치는 영향
  • 활성 시점: Phase 3
red-team — 적대적 반론
  • 역할: Devil's Advocate 역할. 핵심 전제를 의도적으로 반증하고, 역 시나리오를 구성
  • 검증 방식:
    • 핵심 전제 반증: "이 전제가 틀렸다면?"
    • 역 시나리오: 결론의 정반대 상황이 발생할 조건 도출
    • 숨겨진 가정 노출: 분석에 암묵적으로 깔린 미검증 가정 식별
  • 활성 시점: Phase 3

리뷰 에이전트 (Phase 3.7) — 1개

external-reviewer — 프레이밍 비판
  • 역할: 전체 분석의 프레이밍과 접근법을 비판적으로 검토. Red Team이 개별 Claim 단위라면, External Reviewer는 분석 전체 수준
  • 약점 탐지 체크리스트 (5항목):
    • 확증 편향(Confirmation Bias): 우호적 정보만 수집하지 않았는가
    • 반증 부족: 충분한 반대 증거를 검토했는가
    • Groupthink: Division 간 동질적 결론에 빠지지 않았는가
    • 관점 고정(Anchoring): 초기 가설에 과도하게 고착되지 않았는가
    • 대안 부족: 충분한 전략 대안을 검토했는가
  • 외부 모델 연계: 선택적으로 Codex, Gemini 등 외부 모델의 리뷰를 수렴 가능
  • 산출물: thinking-loop/self-critique.md, thinking-loop/external-review.md
  • 활성 시점: Phase 3.7

합성 에이전트 (Sync R2, Phase 3) — 2개

cross-domain-synthesizer — Division 간 교차 분석
  • 역할: 서로 다른 Division의 분석 결과를 교차하여 시너지와 충돌을 식별
  • 분석 관점: Market 기회 vs Capability 현실, Product 전략 vs Finance 투자 여력, 조직 역량 vs 시장 요구 등
  • 산출물: Cross-domain 정합성 보고서, 시너지/충돌 매트릭스
  • 활성 시점: Sync Round 2
insight-synthesizer — 도전 결과 통합
  • 역할: Phase 3 사고 루프(logic-prober, strategic-challenger, red-team)의 결과를 통합하여 전략을 보강하거나 수정
  • 수렴 판정: 도전과 반론이 충분히 반영되었는지 판단. 미해소 긴장(Unresolved Tension)은 보고서에 명시적으로 포함
  • 산출물: 통합 인사이트 보고서, 전략 수정 권고, 수렴/발산 판정
  • 활성 시점: Phase 3 후반

보고 에이전트 (Phase 4) — 2개

report-writer — 세로형 보고서 생성 (Phase 4-A)
  • 역할: 전체 리서치 결과를 경영진 대상 세로형 보고서(report-docs.md)로 변환
  • SCR 스토리라인: Situation(현황) → Complication(문제) → Resolution(해결) 구조로 스토리 구성
  • Action Title: 모든 섹션 제목을 주장 문장형으로 작성 (주제형 금지)
  • Implementation Playbook: 전략 제안에 담당/마일스톤/KPI/의존성을 포함한 실행 계획
  • 활성 시점: Phase 4-A
brief-writer — 경영진 원페이퍼 (Phase 4-C, 선택적)
  • 역할: report-docs.md를 1~2페이지 BLUF 구조로 압축한 독립적 의사결정 문서
  • BLUF: 핵심 결론 1문장(15단어 이내) + 근거 2줄을 문서 최상단에 배치
  • 구성: Key Findings 3개 + Financial Snapshot + Recommended Actions(3~5개, 담당/90일 마일스톤) + Risk Alert
  • 활성 시점: Phase 4-C (Interactive/Team 기본, Auto는 Client Brief 명시 시)

QA 에이전트 (Phase 5) — 5개

qa-orchestrator — QA 파이프라인 조율
  • 역할: Phase 5 QA 파이프라인을 순차 실행. 3개 검증 에이전트(audience-fit, executability, report-auditor)를 스폰하고 결과를 취합
  • 게이트 판정: Critical/Major 이슈 0건 = PASS. 이슈 발견 시 report-fixer 자동 스폰
  • 활성 시점: Phase 5
audience-fit-checker — 경영진 적합성 (6 Check)
  • 역할: 보고서가 경영진 독자에게 적합한지 6가지 항목을 검증
  • 체크 항목: Action Title 적절성, SCR 스토리라인 완결성, Executive Summary 품질, 가독성(길이/구조), 용어 적절성, 시각 자료 활용도
  • 활성 시점: Phase 5
executability-checker — 실행 가능성 (5 Check)
  • 역할: Implementation Playbook의 실행 가능성을 5가지 항목으로 검증
  • 체크 항목: 담당자/팀 명시 여부, 마일스톤 현실성, KPI 측정 가능성, 리소스(예산/인력) 현실성, 의존성/선후행 관계 완전성
  • 활성 시점: Phase 5
report-auditor — 논리 완결성
  • 역할: 보고서의 논리적 완결성을 검증. SCR 구조가 일관되는지, 미해소 긴장이 적절히 반영되었는지 확인
  • 검증 대상: Claim-Evidence 연결 완결성, Division 간 결론 정합성, 미해소 긴장의 명시적 서술 여부
  • 활성 시점: Phase 5
report-fixer — 자동 수정 루프
  • 역할: QA에서 발견된 이슈를 자동으로 수정하고, 재검증을 수행
  • 수정 루프: 수정 → 재검증 → 최대 3회 반복. Critical/Major 0건 달성 시 PASS
  • 수정 범위: 문구 수정, 누락 보완, 구조 조정. 분석 내용 자체의 변경은 PM 승인 필요
  • 활성 시점: Phase 5 (qa-orchestrator가 자동 스폰)

유틸리티 에이전트 — 1개

data-preprocessor — 데이터 전처리
  • 역할: 사용자 제공 데이터 및 API 응답을 전처리하여 Division별로 배포
  • 처리 내용: API 응답 정규화, 데이터 타입 변환, 이상치 탐지, Division별 데이터 분배, 정합성 검증
  • 활성 시점: Phase 0~1 (데이터 수집 단계)

프로토콜 상세

출력 포맷: 4-Layer 피라미드

Layer 0: Claim
        "주장/결론" (경영진 결정 가능한 수준)

Layer 1: Evidence
        "그 근거는?" (분석/인사이트, 출처 명시)

Layer 2: Data
        "데이터는?" (수치, 차트, 구체 사례)

Layer 3: Source
        "어디서?" (URL, 인용문, 신뢰도 스코어)

ID 체계

형식: {Division}{SubDomain}-##

예시: MGE-01 = Market-Geography-EastAsia-01

Golden Facts (SSOT)

  • 파일: findings/golden-facts.yaml
  • 쓰기 권한: fact-verifier만
  • 읽기 권한: 모든 에이전트
  • 역할: 수치 검증의 단일 출처

  • C (Complication): 문제, 기회, 긴장, 교착 상태
  • R (Resolution): 해결책, 실행 경로, 기대 효과

Action Title

  • 모든 보고서 섹션 제목은 주장 문장형 (주제형 금지)
  • 나쁜 예: "가격 전략" → 좋은 예: "프리미엄 가격 전략으로 3년간 40% 마진 달성 가능"

검증 체계

4단계 검증 (VL-1 ~ VL-3 + QA)

VL-1

Leaf 자가 검증

  • 2개 이상 독립 소스 확보
  • 반증 검토 (반대 의견 존재 여부)
  • Confidence 스코어 태깅
VL-1.5

Lead 삼각 검증

  • Leaf 간 교차 검증 (동일 영역 3개 이상)
  • 스팟체크 3건 이상 (출처 직접 확인)
VL-2

Division 정합성

  • 엔터티 정합성 (동일 기업/시장을 같은 이름으로)
  • 시점 정합성 (과거/현재/미래 명확 구분)
  • 정의 일관성 (지표 산출 방식 동일)
VL-3

교차 검증 (fact-verifier)

  • Division 간 모순 탐지
  • Golden Facts 도출 (수치 SSOT)
  • 신뢰도 스코어 매기기

QA Phase 5: 보고서 검증 (8+1단계)

  • 내장 4단계: 구조/링크/포맷/메타데이터
  • 외장 3단계: audience-fit-checker / executability-checker / report-auditor
  • 수정 루프: report-fixer (최대 3회)
  • 합격 기준: Critical/Major 이슈 0건
